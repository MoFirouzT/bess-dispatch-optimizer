## R2.2c IMPLEMENTED (gate green): residual-load-conditional scenario tail (2026-07-24)

The payoff of the forecasting-challenges thread: R2.1c put residual load in the forecaster, R2.2b un-capped spikes but spread them *uniformly*, and R2.2c ties the tail to residual load so spikes land on tight-margin hours. Spec drafted, approved (one design fork put to the human via a question), implemented to green in one session. **Spec: [`docs/specs/R2.2c-conditional-tail.md`](specs/R2.2c-conditional-tail.md) (Implemented); [ADR-0026](decisions/0026-conditional-scale-tail-channel.md).**

- **Design fork resolved by the human: the magnitude channel `β(x)`** (over frequency `λ(x)` or both). Condition the GPD *scale* on residual load via a log-link `β(x) = β₀·exp(γ·z)`; `ξ`/`β₀` reused from R2.2b's unconditional PWM fit (`β₀` = scale at `z=0`), `γ` fit by **OLS of log(excess) on standardized residual load**, clamped `γ ≥ 0`. Clean `γ=0 ⇒ byte-identical R2.2b` identity; pure-numpy (no scipy). Frequency channel deferred to **R2.2d** (named, on record) as the more decision-relevant but larger redesign with no clean identity.
- **New in `bess.scenarios.tail`:** `log_scale_slope` (raw OLS slope), `ConditionalTailModel` (`.fit`, `.beta_at`); `apply_tail`/`generate_scenarios` gain a per-hour `tail_covariate` (per-element GPD scale; `gpd_quantile` now accepts an array β). `ConditionalTailModel` exported.
- **Live result (NL Feb–Jun 2024, verified 2026-07-24):** **`γ = 0.21 > 0`**, so residual load genuinely predicts spike magnitude on this asset. Tail scale rises from **`β ≈ 6.2` on slack hours (10th-pct residual load) to `β ≈ 10.4` on tight hours (90th-pct), about +69%**. This is the thread's thesis confirmed end-to-end: spikes are residual-load-extreme events, and conditioning on residual load makes the scenario set represent that. A no-signal window would fit `γ≈0` and reduce to R2.2b (reported null, not failure).
- **Tests (tests-first):** `tests/golden/test_golden_conditional_tail.py` (OLS slope exact `1`/`2`; `β₀`=R2.2b PWM; β rises with covariate; `γ<0` clamp; `γ=0` identity), `tests/property/test_conditional_tail.py` (opt-in identity, monotone-in-covariate spikes, `γ≥0` clamp, no-signal `γ≈0`, determinism), token-gated `tests/integration/test_conditional_tail_live.py`. Figure `docs/figures/example-conditional-tail.svg` (`plot_conditional_tail` + `examples/conditional_tail_demo.py`); viz + example smoke.
- **No `formulation.md` section** (same call as R2.2b: statistical scenario-layer extension, 600-line cap). Theory in the spec + `references.md` §R2.2c (Coles 2001 **ch. 6 "Extremes of Nonstationary Sequences"** governing, chapter verified against the publisher listing; Hosking-Wallis PWM for the base fit).
- **Gate status:** full suite **229 passed, 19 skipped**; ruff/format/mypy(43)/lint-imports(4 KEPT)/docs-lint clean. R2.2/R2.2b/R2.3/R2.5 unaffected (byte-identical default path). Uncommitted.

## R2.2b IMPLEMENTED (gate green): extreme-value (peaks-over-threshold) tail for the scenario bootstrap (2026-07-24)

Continued the forecasting-challenges thread (the #2 spike-tail work R2.1c unblocked). Spec drafted, approved, and implemented to green in one session. **Spec: [`docs/specs/R2.2b-spike-tail.md`](specs/R2.2b-spike-tail.md) (Implemented); [ADR-0025](decisions/0025-semiparametric-gpd-scenario-tail.md).**

- **The gap closed (#2 of the forecasting-challenges audit):** the R2.2 residual-path bootstrap can only replay historical errors, so the worst spike any scenario contains is capped at the historical-max residual, leaving the R2.3 CVaR tail blind to unprecedented spikes. R2.2b adds a **semiparametric tail**: empirical body (unchanged), Generalized Pareto tail over a high threshold (un-capped).
- **Design (ADR-0025): splice in place, not resample or re-model.** Keep the whole-day bootstrap (intra-day shape, equiprobable, same `n`); replace each residual component's *excess over the threshold* with a fresh GPD draw. Exceedance *frequency* stays empirical (~5% at the 95th-pct threshold), only *magnitude* becomes parametric. Opt-in: `generate_scenarios(..., tail=None)` byte-identical to R2.2 (tail draws come from the same RNG *after* the resample indices).
- **Pure numpy, no scipy.** GPD fit by probability-weighted moments (Hosking-Wallis 1987, closed-form, verified: `ξ̂ = 2 − a0/(a0−2a1)`, `β̂ = 2a0a1/(a0−2a1)`; exponential-excess sanity gives ξ=0/β=mean). New `bess.scenarios.tail` (`fit_gpd_pwm`, `gpd_quantile` inverse-CDF, `TailModel.fit`, `apply_tail`); `tail` threaded through `generate_scenarios`.
- **Live result (NL 2024, held-out days, verified 2026-07-24):** fraction of realized prices above the scenario set's support ceiling (max priceable spike) falls **7.4% (capped bootstrap) → 1.0% (GPD tail)** (ξ=0.27). The tail covers spikes the plain set gave zero probability. *Measured finding:* the *body* 99th-pct coverage is dominated by the crude point forecast, not the tail, so the **support ceiling** (where the cap bites) is the honest metric, not the body quantile the draft proposed; spec gate updated to match.
- **Open question 1 settled by measurement:** forward-selection reduction **retains** the tail (extreme paths sit far from the mass, so dropping one costs Kantorovich distance) → **no tail quota needed**; pinned by `test_reduction_retains_the_tail`.
- **Tests (tests-first):** `tests/golden/test_golden_spike_tail.py` (PWM fit `(ξ,β)=(-1,4)` on `[1,2,3]`; GPD quantile; opt-in + no-exceedance identity; splice-touches-only-exceedances), `tests/property/test_spike_tail.py` (opt-in identity, body-preserved, un-capping, tail-heaviness monotone, valid measure, determinism, reduction-retention, fit-rejects-too-few), token-gated `tests/integration/test_spike_tail_live.py`. Committed figure `docs/figures/example-spike-tail.svg` (`plot_spike_tail` + `examples/spike_tail_demo.py`, synthetic heavy-tail by design; viz + example smoke tests).
- **Formulation: no `formulation.md` section.** Classified with R2.1b (a statistical scenario-layer extension, no optimizer math), and formulation.md is at its 600-line cap (splitting it is a separate task). EVT theory lives in the spec's design sketch + `references.md` §R2.2b (Coles 2001 ch. 4 governing, verified via the publisher listing; Hosking-Wallis PWM subordinate).
- **Gate status:** full suite **217 passed, 18 skipped**; ruff/format/mypy(43)/lint-imports(4 KEPT)/docs-lint clean. R2.2/R2.3/R2.5 unaffected (byte-identical default path). Uncommitted.
- **Deferred (on record):** the *conditional* tail (GPD params as a function of residual load, the richer thing R2.1c unblocks) → R2.2c/future; the lower (negative-price) tail (`TailModel.side` leaves room); README/architecture surfacing (per-phase pattern → milestone).

- **The design nuance STATE.md had flagged ("a MILP has no clean duals") is real but was not the whole problem.** Fix-and-resolve is sound in the *objective* by a containment argument (fixing `u*` restricts the feasible set to a subset containing the MILP optimum, so the LP optimum is exactly equal). The actual trap is that **`u*` is not unique**: at an idle period both `u_t = 0` and `u_t = 1` are optimal and the solver's tie-break is arbitrary, and constraint (3) gates each direction on `u_t`, so the tie-break clamps one direction's cap and moves the reported dual.
- **Measured before drafting, not reasoned about.** On a designed instance (T=3, π=[10,100,200], 1 MW / **2 MWh** so SoC stays interior, e_0=e_tgt=0, η=1) the MILP charges, **idles through the €100 spike**, discharges; objective 190. True `dV/de_0 = 100` by finite difference. Three candidate re-solve rules all return objective 190 and report **μ = 200 / 100 / 10**: only `free_idle` (relax the exclusion cap to the natural power cap at idle periods) recovers the truth. **Objective equality does not discriminate the rules**, which is why oracle 1 carries the gate.
- **A randomized property test cannot catch it:** across 200 random instances all three rules matched the finite-difference truth (random draws pin μ through their trading periods, so the ambiguity never binds). The designed instance is the only discriminator.
- **`free_idle` is unsound as stated, with a closed form.** On 2/600 random instances the relaxed LP **beat** the MILP; both sat at η_rt < 1 with a negative-priced idle period, and the excess matched `Σ |π_t|(1 − η_ch·η_dis)·P̄·Δt` **exactly** (+1.354275, +12.9426). Mechanism: freeing both directions lets the LP charge and discharge at once, holding SoC flat while burning the round-trip loss, and a negative price pays it to. Provably exact at η_rt = 1 (the round trip is then cash- and energy-neutral, so it can be removed without changing feasibility); unproven below 1, hence the guard.
- **The proposed rule:** relax only at idle periods with **π_t ≥ 0**, keep `u*` fixed at negative-priced idle periods, and assert objective equality on every solve. **0 violations across 600 instances**, still μ = [100,100,100] on the reference instance; the fallback engaged on 12 periods, all negative-priced.
- **Two derived claims measured** (500 instances, both η free, negatives, degradation, both power caps): water value **flat on interior SoC** (`μ_t = μ_{t+1}`) **0 violations / 1147**; **no-trade band agrees with the action** `0 / 2503` at pinned periods but **13 / 21** at fallback periods. So the band is only as good as the rule behind it, and at a negative-priced idle period it is *wrong most of the time*, not merely unpinned. That retired the flag-only answer: the proposal now suppresses the band there (open question 2), with a two-tie-break intersection named as the better thing to try first at implementation (unmeasured, so not proposed).
- **A hypothesis the data killed:** "the battery idles at a negative price because it is full, so the honest reason is a binding SoC bound." Measured over 35 such periods: only **2 (6%)** sit at e_max, 18 at e_min, 15 strictly inside. Physical blocking is not the explanation; the binding constraint is often intertemporal (a later discharge cap, or wear making the round trip unprofitable). Recorded in the spec so it is not re-derived.
- **The math (derived from R1.1/R1.2 stationarity, no new variable or constraint):** μ_t = dual of the SoC balance = marginal value of stored energy; flat while SoC is interior; the **no-trade band** `η_ch(μ_t − c_deg) ≤ π_t ≤ (μ_t + c_deg)/η_dis` is the explanation, its width created by round-trip loss and wear (collapsing to the point π_t = μ_t at η=1, c_deg=0), not by the price.
- **Scope calls proposed:** R2.3's two-stage duals **out** (a different economic object: the first stage is a DA commitment coupled by a budget, needing its own delta and oracles), so `Depends on` is R1.1/R1.2/R1.5 and `explain` does not import `stochastic`; slippage/transaction costs **out** (an R1.1 objective change, unrelated to duals, would reopen R1.1/R1.2 oracles); no governing reference (LP duality is standard technique, CLAUDE.md §1 as relaxed 2026-07-13); a separate `POST /explain` **without** the R1.5 breaker, since the breaker's greedy fallback is a heuristic with no duals at all.
- **Open question 1 resolved (2026-07-16): `free_idle` restricted to π_t ≥ 0, `u*` kept at negative-priced idle periods, objective equality asserted every solve.** Shapes ADR-0023. Resolved by measuring the one rival that could have beaten it, not by argument.
    **The rival, and why it lost.** `u_t` is binary, so the idle set admits exactly two tie-breaks, and each is sound alone (neither frees both directions at one period, so neither can dump). Intersecting them looked like it would pin the value soundly and dissolve *both* the π ≥ 0 restriction and the fallback. **Measured: it cannot be read off a solver.** On the reference instance both tie-breaks return the correct objective 190 and report **μ = 200** (u=0) and **μ = 10** (u=1); the truth (100) lies strictly between and equals neither. A solver returns **a vertex of the dual optimal face, not the face**: the faces are `[100,200]` and `[10,100]` and *do* intersect at `{100}`, but HiGHS hands back the far ends, discarding exactly what the intersection needs. Recovering the faces costs ~`4T` auxiliary LPs plus an explicit dual.
    **And the rival is redundant where `free_idle` applies, which is the real finding.** The two reduced-cost conditions at an idle period invert to closed-form bounds **in the price alone**: `π_t·η_dis − c_deg ≤ μ_t ≤ π_t/η_ch + c_deg`. Each tie-break clamps one direction's cap, destroying that direction's condition, so it can impose only one side; `free_idle` frees both and imposes both at once. Measured over 400 instances / 205 idle periods: tie-break u=0 honours the lower bound **0/205**, u=1 the upper **0/205**, `free_idle` **both** (0/203 each). So **`free_idle` *is* the intersection, computed in one LP**, and its dump is the price of getting it in one: the relaxation that pins μ from both sides is the same one that opens the primal to simultaneity. The intersection survives only as open question 2's tool (the sound way to pin a value where `free_idle` may not go), not as a rival rule.
    **What would overturn it:** the LP beating the MILP under the restricted rule (the assertion catches this in CI), or a π ≥ 0 idle period whose reported μ_t misses a two-sided finite difference.
- **Open question 2 resolved (2026-07-16): report μ everywhere; report the band iff the period's flat RUN is pinned; detect pinnedness by solving the fallback tie-break both ways and comparing μ.** Also ADR-0023. Beat the earlier "suppress every fallback band" fallback by measuring.
    **The decision table (81 fallback = negative-priced idle periods).** Cross-tabbing pinned (both fallback tie-breaks agree on μ) × band-empty (closed form) × band-holds: **pinned & band-holds 35, unpinned & fails 25, unpinned & empty & fails 18, unpinned & holds-by-luck 3. No row is pinned-and-failing** → pinnedness is a *sufficient* condition to report the band (35/35). So blanket suppression (the prior proposal) needlessly discarded 35 verified-correct bands. The 3 luck-holds stay suppressed (nothing distinguishes them). Empty band never holds (sanity 0).
    **`pinned` is per-RUN, not per-period (110/110).** μ is constant while SoC is interior, so a tie-break shifts a whole flat run's level. Measured constant within every run. Mechanism, exact: a run containing a trading period dispatching **strictly between 0 and its cap** is pinned **76/76** (its reduced cost is an equality; a period *at* its cap gives only a one-sided bound, which is why the reference instance stays ambiguous). So the interface carries a `FlatRun` (μ + pinned) and `PeriodExplanation.run` indexes it; the drafted per-period `pinned` bool was wrong.
    **Detection is one extra LP for the whole horizon** (only when a fallback period exists), assumption-free: both tie-breaks sound in every instance (0 unsound), and where they agree the tie-break never mattered. This fully retires the dual-face intersection (which cost ~4T LPs for the same safety). Of 38 fallback periods, tie-break didn't matter for 17 (45%) → honest fallback rate ~half the period count.
    **Empty band = wear, not price, holds the battery still.** Band inverts at π<0 unless `c_deg ≥ |π|(1/η_ch − η_dis)/2`; 14/59 fallback periods empty, the 45 non-empty all far above threshold (c_deg 14–18 vs 0.1–2.9). An empty band *proves* the idle is non-economic → goes in the `reason` string, but doesn't gate the decision (every empty case is already unpinned).
    **What would overturn it:** a fallback period where both tie-breaks agree yet the band fails (the band-consistency property catches it).
- **Open question 3 resolved (2026-07-16): no, R2.4 does not explain R2.3's two-stage program.** A scope call, not a measurement. The R2.3 first-stage dual prices the recourse budget `|g^(s) − g^DA| ≤ ρP̄` (plus CVaR's own multipliers z_s, η): a different economic object from the SoC water value, needing its own delta and oracles. Keeps `Depends on` = R1.1/R1.2/R1.5 and `explain` importing only optimizer/assets (the `explain → stochastic` layering edge stays unused-but-permitted; R2.4 could have shipped before R2.3).
- **Open question 4 resolved (2026-07-16): slippage/transaction costs IN scope as a band analysis, OUT as an objective term.** The user's push-back corrected a real conflation I'd made. Two different things wore one name: **objective-term** (`−κ·τ_t` in the max) re-optimises to a different schedule and moves objective 40 + the degradation oracles (forbidden by R2.4's gate); **analysis** reads the already-computed schedule + duals and reports per-trade breakeven slippage, changing no objective and re-solving nothing → touches no R1 oracle, carries its own (oracle 7). The read-off ships in R2.4; the model change is its own future formulation delta/phase.
    **The band arithmetic** (derived from the same stationarity as the band, verified consistent): a grid-side fee κ per MWh traded widens the band to `η_ch(μ_t − c_deg) − κ ≤ π_t ≤ (μ_t + c_deg)/η_dis + κ`. **κ enters flat on each side, OUTSIDE the η factors**, *not* like c_deg, which is η-scaled (η_ch·c_deg low side, c_deg/η_dis high side), because a market fee is on grid energy while wear is storage-side throughput. (Exact grid-vs-storage/symmetric modeling pinned at the formulation delta.) Per-trade breakeven slippage = margin clearing the κ=0 threshold: `η_ch(μ_t − c_deg) − π_t` (charge), `π_t − (μ_t + c_deg)/η_dis` (discharge), ≥0 by optimality. Oracle 7 on oracle 1's schedule: charge breakeven 90 (=100−10), discharge 100 (=200−100). Interface gains `PeriodExplanation.breakeven_slippage_eur_mwh`; property: ≥0 at every executed trade. Precedent for folding an additive analysis (not a new phase): the duration axis, ADR-0022.
- **Open question 5 resolved (2026-07-16): a separate `POST /explain`, no circuit breaker, returns schedule + explanation together, 503 on a non-optimal solve.** An API-contract call.     **Why:** a bit-identical run means the market cleared at the same *cent* repeatedly, which is only plausible at a **structural focal point** of the bid stack: €0.00 (the natural zero bid that excess supply collapses onto), or the SDAC floor/cap (which the price cannot cross, so it pins under scarcity). At any other value, continuous cent-quoted prices make an exact repeat negligibly likely, so that is a freeze. Negative prices below zero are a *different regime* (must-run units paying to stay on) and vary continuously (−50.00, −39.79, −27.30 all distinct), which is why "flag negatives" and "flag the day's minimum" both fail.

### R1.4a key modeling decision (implemented)
- Three quantities over the *existing* optimizer: **ceiling** = full-horizon solve (overnight SoC free); **rolling** = per-day solves, each `e0=e_tgt=0` (deterministic agent has no next-day info at the gate ⇒ honest myopic model, intraday-optimal); **greedy** = 20/80 percentile rule (floor, can trade at a loss).
- Provable, gate-able ordering `V_greedy ≤ V_roll ≤ V*` with `0 ≤ V_roll` (greedy can go negative); gap `V*−V_roll` = overnight arbitrage the deterministic agent can't capture → motivates R2. Headline metric `V_roll/V*`.
- No new math (formulation §R1.4 is derived-only). Leakage = decision for day `d` uses only `Π_d` + carried SoC. Sanity band derived from the fixture's own `mean_daily_spread`, not hard-coded.

### R1.3 design (implemented)
- Pre-flight is **pure** over `(prices, spec, dt)`; **accumulates all issues** (no fail-fast); does **not** re-validate the spec (Pydantic already did). `solve()` auto-runs `check()` as its first line (fail-closed).
- Catches: empty horizon, non-finite price (w/ index in `field`), non-positive/non-finite `dt`, and **terminal-SoC reachability** `−T·Δ⁻ ≤ Δ ≤ T·Δ⁺` (Δ⁺=η_ch·P̄_ch·dt, Δ⁻=P̄_dis·dt/η_dis), the only solve-time-only check (T from len(prices)).
- Ramp-free reachability is necessary-and-sufficient; with ramp it stays **necessary** (sound filter); ramp-coupled infeasibility left to the solver guard. `solve()` now uses `load_solutions=False` so that residual class returns a termination condition instead of raising on solution load.
- The other §9 nightmares are explicitly *out of scope* for pre-flight (direction → golden/property gates; market rules → R1.4; degradation life → R1.2 monotonicity).
