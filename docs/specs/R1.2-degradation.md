# Spec R1.2 — Piecewise-linear degradation cost

**Status:** Draft (awaiting human review)
**Release:** R1  **Depends on:** R1.1 (deterministic core)

## Objective

Append a convex piecewise-linear **degradation cost** to the dispatch objective so the optimizer trades arbitrage revenue against cell wear — penalizing deep cycling so it stops once the price spread no longer covers the rising degradation slope. Modeled with the convex-combination (λ) method and an SOS2 constraint. Degradation is optional; with none configured the model is R1.1 exactly.

## Formulation reference

Implements [formulation.md § "R1.2 — Piecewise-linear degradation cost"](../formulation.md#r12--piecewise-linear-degradation-cost-draft--awaiting-review) in full. **No math outside that section.** Degradation is a cost subtracted from revenue — never in the SoC balance, and it does not add an efficiency term to the objective's cash flow.

## Degradation measure (summary)

Per-period **storage-side throughput**, counting **both directions**:

```
τ_t = η_ch · p_charge_t · dt  +  (p_discharge_t / η_dis) · dt
τ_max = max(η_ch · p_charge_max · dt,  p_discharge_max · dt / η_dis)
```

A round trip of depth `q` is therefore penalized twice — once on the charging period, once on the discharging period. Efficiency appears here because `τ_t` is cell-side energy (a wear quantity), not cash flow.

## Parameters / configuration

A new `DegradationSpec`, optionally attached to `BatterySpec` (default `None` ⇒ R1.1 behavior):

| Field | Meaning | Unit |
|---|---|---|
| `throughput_pu` | breakpoints `φ_0=0 < … < φ_K=1` on per-period storage-side throughput, **per-unit of `τ_max`** (consistent with ADR-0009) | p.u. |
| `cost_eur` | `g_0=0 ≤ … ≤ g_K`, degradation cost at each breakpoint; must be **convex** (non-decreasing slopes) | € |

Validation (Pydantic v2): equal lengths ≥ 2; `throughput_pu` starts at 0, ends at 1, strictly increasing; `cost_eur` starts at 0, non-decreasing, convex. Absolute breakpoint throughput is `φ_k · τ_max`, with `τ_max` computed by the asset from the spec + `dt`.

**Decision (per-unit of `τ_max`):** breakpoints are per-unit of the max per-period storage-side throughput, mirroring ADR-0009 — size- and `dt`-independent, and the top breakpoint always covers the feasible throughput range without coupling config to `dt` or `η`. Absolute-MWh breakpoints were rejected (re-couples to `dt`/`η`).

## Golden oracles (exact expected values)

All use a 1 MWh / 1 MW battery, `dt=1`, `e_0=e_tgt=0`. Tolerance `1e-6`.

| # | inputs | expected objective | expected schedule | why this case |
| --- | --- | --- | --- | --- |
| 1 | `T=2`, `π=[0,50]`, `η=1`, `φ=[0,0.5,1]`, `g=[0,5,35]` | **15.0** | charge `[0.5,0]`; discharge `[0,0.5]`; soc `[0.5,0]` | degradation bites: both-direction cost `2·g`, convex slope (60) > spread (50) past the kink ⇒ cycle stops at `q=0.5` |
| 2 | `T=2`, `π=[0,50]`, `η=1`, `φ=[0,0.5,1]`, `g=[0,1,3]` | **44.0** | charge `[1,0]`; discharge `[0,1]`; soc `[1,0]` | cheap degradation (slopes 2, 4 ⇒ `2·slope < 50`): full cycle still optimal, profit shaved by `2·g(1)=6` |
| 3 | `T=2`, `π=[0,50]`, `η_ch=1, η_dis=0.8`, `φ=[0,0.5,1]`, `g=[0,6.25,25]` | **12.5** | charge `[0.625,0]`; discharge `[0,0.5]`; soc `[0.625,0]` | **storage-side**: `τ_max=1.25`, throughput is cell-side; a grid-side implementation gives a different answer |
| 4 | `T=3`, `π=[10,50,20]`, `η=1`, `degradation=None` | **40.0** | charge `[1,0,0]`; discharge `[0,1,0]`; soc `[1,0,0]` | regression: degradation disabled ⇒ exactly R1.1 oracle 1 |

**Oracle 1 derivation.** `τ_max=1`, so `x=[0,0.5,1]`. Terminal 0 ⇒ discharge = charge = `q`, and with `η=1`, `τ=q` in both periods. Objective `50q − 2g(q)`: margin `50−2·10>0` for `q<0.5`, `50−2·60<0` for `q>0.5` ⇒ `q*=0.5`, objective `25 − 2·5 = 15`.

**Oracle 2 derivation.** Slopes 2 then 4; `2·slope` is 4 then 8, both `<50` ⇒ discharge to the cap `q=1`; objective `50 − 2·g(1) = 50 − 2·3 = 44`.

**Oracle 3 derivation (storage-side).** `τ_max = max(1·1·1, 1·1/0.8) = 1.25`, so `x=[0, 0.625, 1.25]`, slopes 10 then 30. Charge `q` (price 0): `soc_0 = η_ch·q = q`, charge-side `τ_0 = η_ch·q = q`. Discharge to terminal 0: `p_dis = η_dis·soc_0 = 0.8q`, discharge-side `τ_1 = p_dis/η_dis = q`. Revenue `50·0.8q = 40q`; degradation `2g(q)` (q on `x`). Objective `40q − 2g(q)`: margin `40−2·10>0` for `q<0.625`, `40−2·30<0` above ⇒ `q*=0.625`, `p_dis=0.5`, objective `40·0.625 − 2·6.25 = 12.5`. (A grid-side throughput would evaluate `g` at `p_ch=q` and `p_dis=0.8q` with `τ_max=1`, giving a different optimum — this oracle fails such an implementation.)

## Property tests (invariants for any valid input)

- **All R1.1 invariants still hold** (SoC bounds, power caps, mutual exclusion, exact continuity, terminal, objective floor) — degradation touches none of them.
- **Degradation non-negative:** `D_t ≥ -ε` every `t`.
- **PWL evaluator monotone + convex:** for random valid `DegradationSpec` and `0 ≤ x1 ≤ x2 ≤ τ_max`, `cost_at(x1) ≤ cost_at(x2)` (monotonicity gate) and the chord test for convexity.
- **Objective consistency (with degradation):** solver objective `≈ Σ_t π_t·dt·(p_dis−p_ch) − Σ_t cost_at(τ_t, τ_max)`, where `τ_t = η_ch·p_ch·dt + p_dis/η_dis·dt`.
- **SOS2 adjacency:** at most two `λ_{t,k} > ε` per `t`, consecutive in `k`.
- **Disabled-degradation regression:** for random inputs, `degradation=None` yields the same objective and schedule as the R1.1 solve (within tol).

## Out of scope (do not build here)

- **Rainflow** cycle counting and **calendar aging** — genuinely harder / different in kind (rainflow is path-dependent and non-convex; not a per-period cost). Deferred, not faked.
- **Equivalent-full-cycle** normalization — a cheap variation of the per-period throughput measure, simply not built here.
- Non-convex degradation curves (the SOS2 is kept so they *could* be added later, but R1.2 ships convex).
- Validation layer (R1.3), backtest/real data (R1.4), API (R1.5), all of R2.
