# Spec R1.3 — Pre-flight validation layer

**Status:** Implemented (local gate green)
**Release:** R1  **Depends on:** R1.1 (deterministic core), R1.2 (degradation)

## Objective

Add a **pre-flight checker** that runs *before the solver* and returns **structured, typed errors** instead of solver stack traces or a bare `RuntimeError` on infeasibility.
It catches the predictable, input-driven failure class — empty/invalid horizon, non-finite data, and
**terminal-SoC unreachability** (the "selling energy the battery doesn't have" nightmare) — by an exact algebraic feasibility test, not by
running the `solve`.

## Formulation reference

Implements [formulation.md § "R1.3 — Pre-flight feasibility (derived; no new model)"](../formulation.md#r13--pre-flight-feasibility-derived-no-new-model).
**No new constraints, variables, or objective terms** — the reachability inequalities are *corollaries* of the existing R1.1 SoC balance (1), bounds (2), and power limits (3).
The pre-flight check must match that derivation exactly; if the two ever disagree, the formulation governs.

## Scope — what pre-flight does and does not catch

Pydantic already validates a `BatterySpec` at construction (capacities `> 0`, efficiencies `∈ (0,1]`, the per-unit SoC window).
Pre-flight **does not re-check the spec** — by the time it runs, the spec is already valid.
Its value-add is the class of failures that are only knowable at **solve time**, from the spec *combined with* the run inputs (`prices`, `dt`):

1. **Input hygiene** — things that are not `BatterySpec` fields:
   - empty horizon (`len(prices) == 0`);
   - any non-finite price (`NaN` / `±inf`);
   - non-positive or non-finite `dt`.
2. **Terminal-SoC reachability** — the prize.
  The horizon length `T = len(prices)` is unknown until solve time, so this *cannot* be a spec validator.
  Given `T` periods and the per-period charge/discharge envelopes, is `e^{tgt}` reachable from `e_0` at all?

`validate` **accumulates all issues** (does not fail fast), so one call surfaces every problem with the input.

## Reachability test (summary)

Let `Δ = e^{tgt} − e_0` (absolute MWh; the asset converts the per-unit SoC config).
Per-period SoC increments are bounded by the R1.1 power limits (3) and balance (1):

```text
max charge step  Δ⁺ = η_ch · P̄_ch · dt          (most SoC can rise per period)
max dischg step  Δ⁻ = (P̄_dis / η_dis) · dt       (most SoC can fall per period)
```

Over `T` periods the necessary-and-sufficient (ramp-free) condition is:

```text
Δ > 0  (must charge up):   Δ   ≤ T · Δ⁺      else  TERMINAL_UNREACHABLE_CHARGE
Δ < 0  (must discharge):  −Δ   ≤ T · Δ⁻      else  TERMINAL_UNREACHABLE_DISCHARGE
Δ = 0:                     always reachable (idle is feasible)
```

The endpoint box bounds (`e_0, e^{tgt} ∈ [e_min, e_max]`) are guaranteed by the Pydantic SoC-window validator, so reachability is the only remaining feasibility question for the ramp-free model.
See the formulation section for the derivation.

**Ramp interaction (honest boundary).** With ramp enabled (4), the inequalities above are still a **sound necessary condition** (ramp only
*further* restricts the per-period swing), so a `TERMINAL_UNREACHABLE_*` verdict is always correct. They are **not sufficient** under a tight
ramp — a ramp-coupled infeasibility can pass pre-flight and is then caught by the solver's optimality guard (the existing residual path). This
is documented, not hidden; pre-flight is a fast filter for the provable class, the solver stays the final arbiter.

## Parameters / configuration

None. Pre-flight is pure logic over `(prices, BatterySpec, dt)` — no config, no I/O, no solver.

## Interfaces

```python
# src/bess/validation/preflight.py
from enum import Enum
from dataclasses import dataclass

class IssueCode(str, Enum):
    EMPTY_HORIZON                  = "empty_horizon"
    NON_FINITE_PRICE               = "non_finite_price"
    NON_POSITIVE_DT                = "non_positive_dt"
    TERMINAL_UNREACHABLE_CHARGE    = "terminal_unreachable_charge"
    TERMINAL_UNREACHABLE_DISCHARGE = "terminal_unreachable_discharge"

@dataclass(frozen=True)
class ValidationIssue:
    code: IssueCode
    field: str                       # "prices", "prices[3]", "dt", "soc_terminal"
    message: str                     # actionable; embeds the numbers
    context: dict[str, float | int]  # machine-readable: required, reachable, horizon, ...

class PreflightError(Exception):
    def __init__(self, issues: list[ValidationIssue]): ...
    issues: list[ValidationIssue]

def validate(prices, spec: BatterySpec, dt: float) -> list[ValidationIssue]:
    """Pure. Returns ALL issues (possibly empty). Never raises, never solves."""

def check(prices, spec: BatterySpec, dt: float) -> None:
    """Raise PreflightError(issues) if validate() finds any; else return None."""
```

**Integration (proposed — see Open questions):** `optimizer.core.solve()` calls `check()` as its first line, before `build_model()`. The
predictable infeasibility class then surfaces as a structured `PreflightError` and the solver is never invoked; the existing
`termination_condition` guard remains for the residual (e.g. ramp-coupled) class. `Schedule` is unchanged.

**Layering:** `validation` enters the import-linter chain **between `optimizer` and `assets`**:
`api → explain → stochastic → recourse → optimizer → validation → assets`. `validation` imports `assets.battery` (for `BatterySpec`);
`optimizer` imports `validation`. (It was previously left unconstrained; this phase places it.)

## Property tests (invariants for any valid input)

- **Total / pure:** `validate(prices, spec, dt)` never raises and always returns a `list[ValidationIssue]`, for arbitrary `prices` (incl.
  `NaN`/`inf`/empty), arbitrary `dt` (incl. `0`, negative, non-finite), and any valid `BatterySpec`.
- **Soundness — no false positives on reachability (ramp on *or* off):** if `validate` emits `TERMINAL_UNREACHABLE_*`, then `solve` does **not**
  reach optimality on the same input (the model really is infeasible). Reachability is a necessary condition, so this holds with ramp enabled too.
- **Completeness — ramp-free:** with ramp disabled and otherwise-clean inputs, if `validate` returns **no** reachability issue then `solve`
  reaches optimality (the ramp-free condition is also sufficient).
- **`solve()` integration:** `solve` raises `PreflightError` **iff** `validate` returns a non-empty list; when it does, the schedule is never
  produced (no solver stack trace leaks for the predictable class).
- **No regression on known-good inputs:** for the inputs exercised by the R1.1 / R1.2 property strategies, `validate` returns `[]` — pre-flight
  never blocks a solve the existing gates expect to succeed.

## Out of scope (explicit — do not build here)

- **Ramp-aware reachability.** Pre-flight checks only the ramp-free necessary condition; ramp-coupled infeasibility stays with the solver guard
  (documented above). A reachable-set computation under ramp is deferred until a real asset needs it.
- **The other §9 nightmare scenarios are NOT input-validation and are addressed elsewhere — named here to stay honest about pre-flight's reach:**
  *wrong dispatch direction* → the R1.1/R1.2 **golden + property** gates (the optimizer provably trades the right way); *stale market rules /
  inflated IRR* → the **R1.4** data layer + sanity band; *degradation killing the battery* → the **R1.2** monotonicity/convexity gate. Pre-flight
  owns only the *selling-unavailable-energy* (reachability) and *dirty-input* class.
- **Severity tiers / warnings.** One blocking tier only — every issue is a blocker. A non-blocking "warning" class (e.g. economically-degenerate
  but feasible input) is deferred until one actually appears.
- **FastAPI request validation / HTTP error mapping** → R1.5. **Real-data (ENTSO-E) schema validation** → R1.4. **Circuit breaker / latency
  budget** → R1.5.
- **Surfacing issues in `Schedule`** — `Schedule` is unchanged; pre-flight either passes or raises.

## Open questions (for human review)

1. **Should `solve()` auto-run pre-flight?** *Proposed: yes* — fail-closed with `PreflightError` as the first line of `solve()`, so no caller can
   reach the solver with provably-bad input. Alternative: keep `optimizer` pure and require the `api` layer to call `check()`. Recommendation:
   auto-run (defensive by default); the cost is `optimizer → validation` in the layer chain, which is clean.
2. **Layer placement** — *proposed:* insert `validation` between `optimizer` and `assets` (it needs `BatterySpec`; `optimizer` calls it). The
   alternative (a sibling utility imported by both `api` and `optimizer`) is rejected as it weakens the linear chain. Confirm.
3. **`prices` type** — accept any `Sequence[float]` (matching `solve`), with `NaN`/`inf` detection via `math.isfinite`. Confirm we do **not** want
   to also accept a `pandas`/`numpy` series here (those arrive at the R1.4 data layer, which hands clean floats to the optimizer).
