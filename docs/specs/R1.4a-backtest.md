# Spec R1.4a. Backtest engine + baselines + sanity band

**Status:** Implemented (gate green)
**Release:** R1  **Depends on:** R1.1 (core), R1.2 (degradation), R1.3 (validation)

## Objective

A **walk-forward backtest** over a BE/NL-shaped day-ahead price series that reports three revenue quantities:
a **greedy floor**, the **rolling deployable** value (per-day deterministic MILP), and the **perfect-foresight ceiling** (full-horizon MILP);
plus the headline metric **% of perfect foresight captured**, solve-time stats, and constraint-satisfaction.
Two correctness gates ride on it:
a **leakage assertion** (no decision uses future prices) and a **sanity band** (annualized ceiling per MWh sits in a band derived from the series' own statistics).

The live ENTSO-E loader is **R1.4b**: out of scope here.
R1.4a gates on a **synthetic** in-memory series so the suite runs **token-free**.

## Formulation reference

Implements [formulation.md: "R1.4. Backtest semantics"](../formulation.md#r14-backtest-semantics).
**No new constraints/variables/objective**: the ceiling and rolling values are solves of the existing R1.1/R1.2 optimizer; greedy is a feasibility-preserving heuristic.
The provable ordering `V_greedy ≤ V_roll ≤ V*` (with `0 ≤ V_roll`; greedy can trade at a loss, so no zero floor on it) is a gate.

## Data: synthetic, in-memory

- **Gate D series:**
    a **deterministic synthetic** hourly series (seeded), shaped like a calm NL day-ahead month; cheap nights, evening peak, gentle midday, occasional solar dip.
    Generated by `bess.data.fixtures.synthetic_day_ahead` (seeded), which gate D calls from `tests/golden/test_sanity_band.py`; **nothing is committed to `tests/fixtures/`**.
    This still gates ordering, leakage-magnitude, and sign/efficiency bugs via the self-derived [sanity band](../formulation.md#sanity-band-gate-d).
- **Internal schema (the contract the engine consumes):**
    a `pandas.Series` of `price_eur_mwh`, tz-aware **UTC** `DatetimeIndex` (freq `1h`), gap-free ([conventions §1](../conventions.md#1-time)).
    `bess.data.fixtures.load_prices` validates *this* schema for any parquet a fetch/cache produces; it is unit-tested against an in-tmp synthetic parquet.
    The engine never touches raw ENTSO-E shapes (quarantined to R1.4b's adapter).
- **Real data:**
    ENTSO-E is fetched at runtime (R1.4b, token), never committed; the real-statistics band check is an R1.4b token-gated integration test.
    Why no market data is committed: [ADR-0005](../decisions/0005-commit-real-fixture-data.md).
    A live probe earlier confirmed the engine on real NL 2024: ceiling ≈ €27k/MWh-yr in the [sanity band](../formulation.md#sanity-band-gate-d), rolling ≈ 99.7% of perfect foresight. Reproduce via R1.4b, not from committed data.

### The greedy policy (precise, feasibility-preserving)

For each window's price block, with `p20`/`p80` the percentiles (pinned `method`):

1. **Charge**
    in period `t` iff `price[t] ≤ p20` **and** some later period `t'` in the same window has `price[t'] ≥ p80` **and** unclaimed discharge capacity remaining at `t'` (its power cap minus what earlier charges have already earmarked for it) to sell this period's energy into;
    charge at the most the SoC headroom + power cap allow.
2. **Discharge**
    in period `t` iff `price[t] ≥ p80` and SoC permits;
    discharge at the most SoC + power cap allow.
3. **Otherwise idle.**
    A charge is additionally capped so the *trailing* periods can fully discharge it;
    any SoC left at window end is liquidated by force-discharging from the **final period backward** (keeping SoC ≥ `e_min` throughout) so the window **ends empty**: preserving the floor property `V_greedy ≤ V_roll`.
    Greedy respects all R1.1 physics (power caps, SoC bounds, efficiency in the SoC balance, mutual exclusion) by construction; it is checked against the same property invariants, since it is **not** solver-produced.

> The exact tie-break / liquidation procedure is frozen by the golden oracles below; the implementation must reproduce them.

## Golden oracles

1 MWh / 1 MW asset, `dt=1`. Tolerance `1e-6`.
Each "day" below is one window.

| # | inputs | greedy | rolling | ceiling | % captured | what it pins |
| --- | --------------- | -------- | --------- | --------- | ----------- | -------------- |
| 1 | 1 day <br> `π=[40,42]`, `η=0.95` | **−2.095** | **0.0** | **0.0** | n/a (`V*=0`) | greedy < rolling: greedy ignores the round-trip breakeven (5% spread < ~10.8%) and **trades at a loss**; MILP idles |
| 2 | 2 days <br> `π=[10,40  50,5]`, `η=1` | **30.0** | **30.0** | **40.0** | **75%** | rolling < ceiling: ceiling carries SoC **overnight** (buy@10 d1 → sell@50 d2, +40); rolling commits per-day (best is buy@10/sell@40 = 30, d2 no feasible trade) |
| 3 | 2 days flat <br> `π=[20,20, 20,20]`, `η=1` | **0.0** | **0.0** | **0.0** | n/a | no spread ⇒ all three idle; degenerate `% captured` handled (`V*=0 ⇒ report 0`) |

**Oracle 1 (greedy loss).** `p20(method=linear)` of `[40,42] = 40.4`, `p80 = 41.6`. Greedy charges @40 (≤40.4), discharges @42 (≥41.6); grid-side it pays `40` to store `0.95`, sells `0.9025` @42 = `37.905`, net **−2.095**. The MILP sees `42/40−1 = 5% < 1/η^{rt}−1 ≈ 10.8%` and idles ⇒ rolling = ceiling = **0**. Pins that optimization *avoids* losing trades a naive rule makes.

**Oracle 2 (overnight value).** Ceiling solves `[10,40,50,5]` full-horizon, `e0=e_end=0`, η=1: buy@10, hold overnight, sell@50 ⇒ **40**. Rolling: day1 `[10,40]` → buy@10/sell@40 = **30**, end empty; day2 `[50,5]` → to start/end empty the only ordering is sell-then-buy (no energy to sell first) ⇒ **0**. So rolling = **30**, captured = 30/40 = **75%**, gap **10** = overnight arbitrage. Greedy day1 = buy@10/sell@40 = 30; day2 no later high after the low ⇒ idle ⇒ greedy = **30**.

## Property tests

- **Provable ordering:** `V_greedy ≤ V_roll + ε` and `V_roll ≤ V_ceiling + ε`, for random multi-day price series + valid specs (the core formulation gate of this phase).
- **Constraint satisfaction (all three baselines):** every concatenated schedule satisfies the R1.1 invariants; power caps, mutual exclusion, SoC bounds, **exact** SoC continuity (efficiency grid-side), each window ends empty. Reuses the R1.1 property checks; especially guards **greedy** (not solver-produced).
- **Leakage (gate C):** perturbing prices strictly after window `i` leaves window `i`'s rolling schedule **unchanged** (decisions use only `Π_d` + carried SoC). Structurally guaranteed by the windowing, asserted directly.
- **Objective consistency:** each baseline's reported `revenue_eur` equals the grid-side cash flow recomputed from its own schedule (`Σ π·dt·(p_dis − p_ch)`), no efficiency term.
- **Metric well-formed:** `pct_of_perfect_foresight ∈ [0, 1+ε]`; defined as `0` when `V* == 0`.

## Duration as a reported axis (ADR-0022)

The optimizer is scale-invariant in the asset's power and energy ratings, so `run_backtest` is correct at any energy-to-power ratio (the property tests already exercise `capacity ≠ 1`).
But the *reported* economics are duration-dependent, and the headline conclusions move across the 1h–4h range: `pct_of_perfect_foresight` falls as duration rises (cross-day carry grows), and `annualized_ceiling_per_mwh` falls (diminishing returns per MWh installed).
A single-duration headline can therefore misstate the general result; see [ADR-0022](../decisions/0022-storage-duration-reported-axis.md).

- **Requirement:**
    capture ratio and per-MWh economics are reported across a small duration set (default **{1h, 2h, 4h}**, power fixed), as a table or curve, not for a single asset.
    A single-duration figure names its duration.
- **Sanity-band caveat:**
    the gate-D band uses `mean_daily_spread` (max − min per day), a **1-hour-asset** notion (it credits the single cheapest / most-expensive hour).
    A duration-`D` asset charges and discharges across its `D` cheapest / most-expensive hours, a *narrower* effective spread, so the 1h band over-predicts its per-MWh ceiling.
    Gate D therefore stays on the **1h reference asset**; the sweep reports capture ratios without asserting the 1h band on longer-duration assets.
    A duration-aware band (top-`D`-minus-bottom-`D`-hour spread, cycles/day per duration) is the correct generalization if a band gate on non-1h assets is ever wanted.

## Out of scope

- **Live ENTSO-E loader / `entsoe-py` client, 15-min native** → **R1.4b**.
- **ELIA / imbalance / intraday data** → later; this is day-ahead arbitrage only.
- **Forecast-driven dispatch, overnight SoC carry via a forecast, recourse** → R2 (the ceiling gap is precisely what motivates it; do not fake it here).
- **Reporting visuals / README hero charts** → packaged at Release-1 ship, not in this gate.
- **Degradation in the backtest objective**: supported (R1.2 is additive) but the sanity band/oracles here use the R1.1 asset; degradation-on backtests are a follow-up.
- **Additional baselines (future work; decide later).** The current greedy is a deliberately weak *within-day* floor: it sees the whole day-ahead block (percentiles + within-day lookahead), so `V_greedy ≤ V_roll` is a heuristic-vs-optimal gap at equal information, not a foresight gap. Two candidate comparators, neither built here, are worth considering when a stronger benchmark is wanted:
  - a **causal / online threshold policy** (decide each hour with no within-day lookahead) as a *true* myopic floor that reflects an information disadvantage rather than heuristic weakness;
  - a **stronger information-limited baseline** (backward-induction DP, or MPC-with-forecast) as a tighter lower comparator sitting between greedy and rolling.
  Both are evaluation baselines only; they add no optimizer math. Relates to resolved decision 4 below.

4. **Greedy liquidation rule.**
    **Resolved:** the greedy baseline is the feasibility-preserving percentile rule over empty→empty windows (`backtest.baselines`), the intended `V_greedy` floor beneath rolling and the perfect-foresight ceiling.
