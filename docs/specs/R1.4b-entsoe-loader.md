# Spec R1.4b — Productionized ENTSO-E data loader

**Status:** Approved — frozen for implementation (open questions resolved below)
**Release:** R1  **Depends on:** R1.4a (backtest engine + fixture schema)

## Objective

A productionized `bess.data` **ENTSO-E day-ahead loader** that produces the *same internal price-series schema* R1.4a already consumes (`load_prices`), pulling real BE/NL history directly from the ENTSO-E Transparency Platform (token-gated), with on-disk caching. Then a **real-data sanity-band re-validation** (token-gated integration test): confirm the §5 band on real Q1 vs. a volatile summer slice (band shifts up) and the ordering on real data. CI stays on R1.4a's synthetic gate — **no real price data is committed**; ENTSO-E is the runtime/integration source only.

## Verified API facts (probed live this session — not from memory)

The guardrail (`CLAUDE.md` §7) fetch is **done**; coding targets this confirmed shape:

- **Host:** `https://web-api.tp.entsoe.eu/api` — *note `tp`, not `tps`*; the old `web-api.tps.entsoe.eu` no longer resolves. Auth: `securityToken` query param. Rate limit: 400 req/min/IP.
- **Day-ahead prices:** `documentType=A44`, `in_Domain`/`out_Domain` = the zone EIC (NL `10YNL----------L`, BE `10YBE----------2`); `periodStart`/`periodEnd` = `YYYYMMDDHHMM` UTC.
- **Response:** `Publication_MarketDocument` → one **`TimeSeries` per market day** → `Period` with `resolution` (`PT60M` for 2024 NL; `PT15M` after the 2025-10 switch) → `Point`s with **1-based `position`** + `price.amount`; `currency_Unit.name=EUR`, `price_Measure_Unit.name=MWH`; `timeInterval` in **UTC** (`...T23:00Z` = CET midnight).
- **Curve quirk (must handle):** ENTSO-E A03 curves **omit a `Point` when the price is unchanged** — a missing `position` carries the previous price forward. The loader must expand positions to a full regular series (entsoe-py does this).
- **Feed sanity:** the probed NL prices (0.1, 0.01, 0.0, …) match published EPEX/ENTSO-E day-ahead values → the parser targets the right series. (An earlier Energy-Charts cross-check confirmed identical values before that third-party slice was removed.)

## Environment note (this machine / network)

A TLS-intercepting proxy with a private CA sits on the network: **`curl` trusts it (macOS Keychain), but uv's bundled Python does not**, so raw Python/`requests`/entsoe-py HTTPS to ENTSO-E fails `CERTIFICATE_VERIFY_FAILED`. Verified fix: point Python at the Keychain roots —

```bash
security find-certificate -a -p /Library/Keychains/System.keychain > ca.pem
security find-certificate -a -p /System/Library/Keychains/SystemRootCertificates.keychain >> ca.pem
export REQUESTS_CA_BUNDLE=$PWD/ca.pem   # entsoe-py uses requests
export SSL_CERT_FILE=$PWD/ca.pem        # stdlib urllib
```

This is **operator setup, not code** (documented in the README data section + `.env.example`); the loader must not hard-code a CA path. CI never hits the API, so it is unaffected.

### Licensing (binding)

**Commit no copyrighted price data.** Raw ENTSO-E data is copyright of the Primary Owner of Data, its terms are legally ambiguous, and they grant no public-redistribution right. The repo commits **no real price data at all** (the user's decision to keep it third-party-free): committed test data is **synthetic** (fabricated, no copyright); real ENTSO-E history is *fetched at runtime / in integration tests*, never committed.
