# Spec R1.4c. Anomaly-aware ingestion circuit breaker

**Status:** Implemented (gate green)
**Release:** R1  **Depends on:** R1.4b (ENTSO-E loader, `bess.data`); R1.5 only as a vocabulary cross-reference ([ADR-0013](../decisions/0013-shared-degradation-vocabulary.md)), **not** a build dependency

## Objective

A second, distinct circuit breaker that wraps the **fetch** (not the solve):
it classifies every price-series fetch as `healthy`, `outage`, or `anomaly`, never lets a corrupted-but-present series reach the optimizer unflagged, and falls back to the last-known-good cached series on either failure class.
It lives in `src/bess/data/ingestion_guard.py` as a separate, testable module so the outage-vs-anomaly distinction is unambiguous in code.
This closes the "the feed silently lied to the algorithm" failure mode that a shared dispatch platform is most exposed to.

## Formulation reference

**n/a: no new math.**
R1.4c is data integrity and operations over the existing loader. No constraint, objective term, or efficiency placement changes.
The internal price-series schema is the one already enforced by `bess.data.fixtures.validate_price_series` (UTC, sorted, regular, gap-free).

## Design decisions

Both are **Accepted**; the reasoning, alternatives, and failure modes live in the ADRs (Tier 2 canonical), not here.

- [**ADR-0012**: a separate ingestion breaker from the solver breaker](../decisions/0012-separate-ingestion-breaker.md).
    Two distinct breakers with two failure taxonomies, rather than one generic breaker wrapping both fetch and solve, so an operator sees *which layer* failed. Also draws the R1.3-vs-R1.4c line: R1.3 asks *is this problem solvable*, R1.4c asks *can this data be trusted*.
- [**ADR-0013**: one shared degradation vocabulary across both breakers](../decisions/0013-shared-degradation-vocabulary.md).
    The guard emits `GuardResult` from the `data` leaf and consumers compose its `status` with the solver's `mode`, so a solve on stale prices is never reported as plainly healthy. Consumers read the value; the guard never imports upward.

## The anomaly checks

The classifier's job is discrimination, and the sharp domain trap is here:
**zero and negative prices are legitimate** in BE/NL day-ahead markets (high-renewable/low-demand windows; the repo already models a negative-price "paid load", golden oracle 4).
A naive "flag €0.00 / flag negatives" check would misclassify real market conditions as corruption and fall back to stale prices; causing the exact bad dispatch it is meant to prevent, in the opposite direction.
The checks therefore key on *feed pathology*, not on *price level*:

| # | Check | Fires on | Deliberately does **not** fire on |
| --- | ------- | ---------- | ----------------------------------- |
| 1 | **Stuck feed** (shape-aware) | ≥ `max_repeat` consecutive **bit-identical** values at an *arbitrary* price, or ≥ `max_focal_repeat` at a **focal** price | a real low/zero/negative price that still *varies*; **and** a long run at a focal price (€0.00, the band edges), which is market behaviour, not a freeze |
| 2 | **Out-of-band** | value `< band_lo` or `> band_hi` (default `[-600, 5000]` €/MWh) | any value inside the band, including legitimate negatives and zeros |
| 3 | **Non-finite** | any `NaN` / `±inf` in the series | – |
| 4 | **Structural** (surfaced from `validate_price_series`) | timestamp gap in the expected grid, duplicate timestamp, non-UTC / unsorted index, **resolution change** (named separately, see below) | a regular gap-free series |
| 5 | **Empty / short** | zero points; or fewer than `expected_slots_per_day` **when the caller supplies it** (off by default) | a full 24/96-slot day, **and** any legitimate partial-window fetch when the caller has not opted in |

Outage class (transport, before any content check):
timeout, connection error, HTTP 5xx, or an empty HTTP body.
Distinct from anomaly by construction, an outage means *no present data*; an anomaly means *present but untrustworthy data*.

`band_lo`, `band_hi`, and the stuck-feed threshold are configuration, not literals (see below).
The band is **grounded in the EPEX SDAC harmonised clearing-price limits** (currently min −600 €/MWh from 2026-05-28, max 4000 €/MWh base, escalatable by +1000 steps during scarcity), with one escalation step of headroom → default `[-600, 5000]`.
A value outside it *cannot be a real day-ahead clearing price*, so it is corruption by definition (a €99999 spike, a €−9000 sign error).
This is a market technical bound, **not** the revenue-year-specific [sanity band](../formulation.md#sanity-band-gate-d); it does not shift with volatility, only if SDAC itself changes its limits.

## Parameters / configuration

| Item | Where | Default |
| --- | --- | --- |
| `sanity_band` `(lo, hi)` €/MWh | guard config (dataclass, env-overridable for ops) | `(-600.0, 5000.0)`: EPEX SDAC limits + headroom |
| `max_flat_hours` (stuck-feed, **non-focal** price) | guard config | `4.0` wall-clock hours; `max_repeat = ceil(max_flat_hours / dt_hours)` → 4 at 60-min, 16 at 15-min |
| `max_focal_flat_hours` (stuck-feed, **focal** price) | guard config | `24.0` wall-clock hours → 24 at 60-min, 96 at 15-min |
| `FOCAL_PRICE_EPS` | module constant | `0.01` €/MWh: prices are quoted to the cent |
| `expected_slots_per_day` | **caller-supplied**; `None` (the default) disables the `short` check | `None` |
| last-known-good source | the existing `data/cache/*.parquet` (R1.4b), gitignored | per zone/window |

Operational knobs are guard config with env-overridable defaults ([conventions §5](../conventions.md#5-configuration)).
The stuck-feed threshold is expressed in **wall-clock hours**, not slot count, so it survives the 60→15-min resolution switch.

**Why the stuck-feed check keys on the price, not the run length.**
A bit-identical run means the market cleared at the *same cent* repeatedly, and that is only plausible at a **structural focal point** of the bid stack, where the merit order has a genuine flat step.
Two exist: **€0.00**, the natural zero bid that excess supply collapses the clearing price onto; and the **SDAC technical floor / cap**, which the price cannot cross and so pins against under scarcity.
At any other value prices move continuously and are quoted to the cent, so an arbitrary value recurring for hours has negligible probability: that is a frozen feed.

*Measured, not assumed* (NL + BE, full-year 2024).
Every run ≥ 3 h sits at `|p| ≤ €0.01` (8 h and 7 h at 0.00, 3 h at 0.00, 3 h at −0.01), while runs at arbitrary values never exceed **2 h**.
Negative prices below zero are a different regime (must-run units paying to stay on) and vary continuously: −50.00, −39.79, −27.30 are all distinct.
So the discriminator is the **value**, not the length.
Also measured: NL + BE 2025 to the PT15M switch (nothing ≥ 6 h), and NL Nov 2025 at PT15M (longest 0.75 h, at an arbitrary value).

A tempting alternative, *rejected on the data*: "a legitimate flat run sits at the bottom of its day's range."
The 7 h zero run on 2024-04-07 sits at the **33rd percentile** of its day (that day's minimum was −50.00), so position-in-range does not separate the cases; only the value does.

**This strictly dominates the length-only rule it replaces.**
Against the real 2024-03-24 day an 8 h threshold classified genuine prices as `stuck_feed`, and pushing it to 12 h to clear that run left an arbitrary-value freeze undetected for 11 h.
Keying on the value removes the false positive *and* tightens detection: a freeze at €73.07 is caught in **4 h**, three times faster, because the non-focal bound no longer has to admit the zero run.

The token-gated integration test checks both bounds against a **full year** of real NL and BE prices, and asserts (i) the slice is **non-vacuous** (it still contains a long legitimate focal run) and (ii) the premise itself holds on real data, the longest arbitrary-value run being shorter than the longest focal run.
Assertion (i) is load-bearing: the check previously ran on a one-week slice whose longest run was 1 h, so it passed trivially and could never have caught the bad default.

**Known limit.**
The focal allowance is still a length cap, so a focal price is not licensed indefinitely: a full day pinned at exactly €0.00 is an all-zeros feed, not a market, and fires.
Real NL/BE zero runs peaked at 8 h in 2024 and grow with solar buildout, so `max_focal_flat_hours` is the number to revisit if that changes; the integration test measures it.

**Why `expected_slots_per_day` is caller-supplied, not derived.**
Deriving it from the series resolution and always applying it would turn every legitimate **partial-window** fetch into a `short` anomaly: asking for six hours of prices and getting six hours back is a correct answer, not a truncated feed.
The guard cannot tell a deliberately short window from a truncated one, because only the *caller* knows what it asked for. So the check is opt-in, and the caller that does know its window enforces coverage there instead: `fetch_day_ahead` rejects a series that does not span its requested `[start, end]` (R1.4b).
Leaving this off is therefore a deliberate division of labour, not an oversight; do not "fix" it into a false positive.

**A resolution change is not a gap.**
A window straddling an MTU change (the 2025-10 SDAC switch from PT60M to PT15M) is genuinely irregular, so the guard is right to refuse it: the engine cannot consume a mixed-resolution series.
But **nothing is missing**: the feed is complete and correct, the market changed shape, so reporting `schema:gap` sent an operator hunting for timestamps that were never absent.
`classify_series` now names it `schema:resolution_change`: two contiguous segments, each internally regular, at different steps (the real event was 46 hourly steps then 104 quarter-hourly ones), each at least 4 slots long.
The discriminator is deliberately conservative; anything that is not cleanly two long regimes stays a `gap`, and both classify ANOMALY either way, so a misnamed edge case costs a log line rather than a wrong decision.

The **status stays ANOMALY**, deliberately.
The effect is right (do not consume, fall back), and a fourth `FeedStatus` for a once-a-decade market event would ripple through `compose_provenance` and R1.5's shared vocabulary ([ADR-0013](../decisions/0013-shared-degradation-vocabulary.md)) to buy a word.
With a precise reason token the ambiguity that actually costs debugging time is gone.
In the *production* path the loader validates first and raises, so the guard reports `schema:invalid` with the validator's message (which lists the steps seen) as `detail=`; the classifier path is what the token serves.

**Reason tokens carry a diagnosis, not a Python type.**
`reason` is a stable, greppable token and the log carries the underlying `detail`.
The fetch-path schema arm reports `schema:invalid` **plus** the validator's message; it previously used `type(exc).__name__`, which reported `schema:valueerror` for every cause (timezone, gap, duplicate, resolution change, window truncation) and discarded the only text that said which rule fired.
That is precisely the conflation [ADR-0012](../decisions/0012-separate-ingestion-breaker.md) exists to prevent, one level down.

## Interfaces

```python
# src/bess/data/ingestion_guard.py  (data stays a leaf: imports nothing else in bess)
from enum import Enum

class FeedStatus(str, Enum):
    HEALTHY = "healthy"
    OUTAGE  = "outage"
    ANOMALY = "anomaly"

@dataclass(frozen=True)
class GuardResult:
    status: FeedStatus
    prices: pd.Series          # HEALTHY → the fetched series; degraded → last-known-good cache
    reason: str | None         # the specific check that fired, a stable greppable token:
                               #   "stuck_feed" | "out_of_band" | "non_finite" | "empty" |
                               #   "schema:gap" | "schema:resolution_change" |
                               #   "schema:tz" | "schema:duplicate" |
                               #   "schema:unsorted" | "schema:index"      (classifier path)
                               #   "schema:invalid"                        (fetch path; the
                               #     validator's message goes to the log as `detail=`)
                               #   "timeout" | "connection" | "transport"  (outage)
    degraded: bool             # True when status != HEALTHY (a fallback series was substituted)

def is_focal_price(value: float, sanity_band=(-600.0, 5000.0)) -> bool:
    """Is `value` a structural focal point the market plausibly clears at repeatedly?
    Zero (the natural zero bid) or either technical band edge. A bit-identical run at a
    focal price is market behaviour; at any other value it is a frozen feed."""

def classify_series(series: pd.Series, *, sanity_band, max_repeat, max_focal_repeat,
                    expected_slots_per_day) -> tuple[FeedStatus, str | None]:
    """Pure content classifier over an already-fetched series. HEALTHY or ANOMALY only
    (transport/outage is decided before this runs). No I/O, the un-fakeable core.
    The stuck-feed check is shape-aware: each bit-identical run is judged against
    `max_repeat` or `max_focal_repeat` by whether it repeats a focal price. Both are
    resolved slot counts (the config layer converts wall-clock hours / dt)."""

def guarded_fetch(
    fetch_fn,                  # a zero-arg callable performing the real fetch (wraps R1.4b fetch_day_ahead)
    *,
    last_known_good,           # callable | Series | None: the cache fallback
    sanity_band=(-600.0, 5000.0),
    max_flat_hours=4.0,        # non-focal → max_repeat = ceil(hours / dt) from the series freq
    max_focal_flat_hours=24.0, # focal (0.00, band edges) → max_focal_repeat, same conversion
    expected_slots_per_day=None,
) -> GuardResult:
    """Fetch → classify → (fallback + log) → GuardResult. Never raises on a bad feed:
    outage and anomaly both fall back to last-known-good and are logged with the reason.
    Raises only if a fallback is needed and none is available (a genuine hard stop)."""
```

- `classify_series` is pure and side-effect-free so property tests hammer it directly.
- `guarded_fetch` owns the try/except: transport exceptions → `OUTAGE`; `ValueError` from `validate_price_series` → `ANOMALY(reason="schema:…")`; otherwise run `classify_series` on the valid series for content anomalies. Logging distinguishes `outage` from `anomaly` and names the check.

## Golden oracles

| # | input feed | expected `status` / `reason` | why this case |
| --- | ------------ | ------------------------------ | --------------- |
| 1 | clean synthetic day (`synthetic_day_ahead`, 1 day) | `HEALTHY` / `None` | the guard passes good data through untouched |
| 2 | day frozen at an **arbitrary** price (€73.07) ≥ `max_repeat` long | `ANOMALY` / `stuck_feed` | the market does not clear at the same arbitrary cent for hours: a frozen feed |
| 2b | **same-length** block at the **€0.00 focal price** | `HEALTHY` / `None` | the pair to oracle 2, and the shape check's whole point: real NL *and* BE cleared at €0.00 for 8 h on 2024-03-24, so length alone cannot separate these two |
| 2c | a full day (≥ `max_focal_repeat`) pinned at €0.00 | `ANOMALY` / `stuck_feed` | focal ≠ unbounded: an all-zeros feed is not a market |
| 2d | block at the **band edge** (SDAC cap) ≥ `max_repeat` long | `HEALTHY` / `None` | a scarcity pin is structural, not a freeze (principled; not observed in NL/BE 2024) |
| 3 | day with one interior timestamp slot removed | `ANOMALY` / `schema:gap` | structural fault surfaced from `validate_price_series` as a classification, not a crash |
| 3b | window straddling an MTU change (PT60M → PT15M) | `ANOMALY` / `schema:resolution_change` | still refused (unconsumable), but **not** called a gap: nothing is missing, the market changed shape |
| 3c | day with two *adjacent* slots removed | `ANOMALY` / `schema:gap` | the discriminator is conservative: only cleanly two long regimes count as a resolution change |
| 4 | day with a single €9999 point (`> band_hi`) | `ANOMALY` / `out_of_band` | an implausible spike is caught by the wide band |
| 5 | **legitimate** low day: real-shaped negatives/zeros that still **vary** each hour | `HEALTHY` / `None` | the domain trap, a real solar-glut day must **not** be flagged (anti-false-positive) |
| 6 | `fetch_fn` raises a timeout | `OUTAGE` / `timeout`, `degraded=True`, `prices == last_known_good` | transport failure falls back and is labeled outage, not anomaly |

## Property tests

- **No corrupted series ever passes as healthy:** for any injected fault (freeze at an arbitrary price ≥ `max_repeat`, freeze at a focal price ≥ `max_focal_repeat`, gap, duplicate, out-of-band, NaN), `status != HEALTHY`.
- **No false positives:** any schema-valid, fault-free series with arbitrary in-band prices (including negatives and zeros that vary) → `status == HEALTHY`.
- **No false positive on a focal run:** a bit-identical run at €0.00 of *any* length inside `max_focal_repeat` → `status == HEALTHY`. The pair to the first invariant: an arbitrary-price freeze of the same length must fire, and this must not. Real NL and BE both cleared at €0.00 for 8 consecutive hours on 2024-03-24.
- **Outage ⊥ anomaly:** transport-shaped failures classify `OUTAGE`; content-shaped faults classify `ANOMALY`; the two never collide in the logged `status`.
- **Fallback safety:** whenever `status != HEALTHY` and a `last_known_good` exists, `GuardResult.prices` is the fallback (a schema-valid series), never the corrupted one; `degraded is True`.

## Out of scope

- **Fetch inside `POST /dispatch`**: R1.5's client-supplied-prices contract is unchanged; the guard wraps the offline/batch fetch path, not the endpoint.
- **Retry/backoff policy tuning, alerting transports (PagerDuty/Slack/email)**: the guard *classifies and logs*; wiring a real alert sink is ops config, not this phase.
- **Forecast-drift monitoring**: that is R2.1b (a different signal: model decay, not feed corruption).
- **Imputation / auto-repair of a bad series**: the guard falls back to last-known-good; it does not interpolate or "fix" corrupted data (silent repair is itself a nightmare failure mode).
- **A general data-quality framework / dbt-style test runner**: a single focused module; the dbt/software-defined-asset framing is one ADR sentence, not a dependency.

## Resolved decisions (reviewed 2026-07-01)

1. **Stuck-feed threshold**: **resolved:** express as **wall-clock hours** (`max_flat_hours = 8.0`), converted to a slot count per resolution (`ceil(hours/dt)`), not a fixed slot count. A stuck feed is frozen indefinitely, so 8 h catches the pathology without false-positiving on a real flat overnight run, and the hours framing survives the 15-min switch. Longest legitimate flat run to be confirmed against a real slice in the integration test.
2. **Sanity band**: **resolved: `(-600, 5000)`**, grounded in the **EPEX SDAC harmonised clearing-price limits** (min −600 €/MWh from 2026-05-28, max 4000 €/MWh base + one +1000 escalation step of headroom), verified against EPEX/ENTSO-E this session. A market technical bound, not the year-specific revenue band; revalidate only if SDAC changes its limits.
3. **`GuardResult` home**: **resolved: the `data` leaf**, a frozen dataclass, *read* (not imported upward) by consumers exactly as `api` reads `Schedule` from `optimizer`. No new import edge; the "data is a leaf" contract stays KEPT.
4. **Ship-gating**: **resolved: R1.4c gates the Release-1 ship.** The master plan's R1 definition-of-done already includes it (checklist, hero visual #3, commit line); shipping "done" then bolting on a promised visual is weaker than shipping once, complete. R1.5 remains a defensible standalone fallback if an immediate publish is ever forced.
