# Spec R1.4c. Anomaly-aware ingestion circuit breaker

**Status:** Implemented (gate green; four resolved decisions below)
**Release:** R1  **Depends on:** R1.4b (ENTSO-E loader, `bess.data`), R1.5 (circuit-breaker pattern + degradation vocabulary)

## Objective

A second, distinct circuit breaker that wraps the **fetch** (not the solve): it classifies every price-series fetch as `healthy`, `outage`, or `anomaly`, never lets a corrupted-but-present series reach the optimizer unflagged, and falls back to the last-known-good cached series on either failure class. It lives in `src/bess/data/ingestion_guard.py` as a separate, testable module so the outage-vs-anomaly distinction is unambiguous in code. This closes the "the feed silently lied to the algorithm" failure mode that a shared dispatch platform is most exposed to.

## Formulation reference

**n/a: no new math.** R1.4c is data integrity and operations over the existing loader. No constraint, objective term, or efficiency placement changes. The internal price-series schema is the one already enforced by `bess.data.fixtures.validate_price_series` (UTC, sorted, regular, gap-free).

## Design decisions (to be recorded as ADRs)

- **ADR-0012, a separate ingestion breaker from the solver breaker.** Two distinct breakers, two failure taxonomies, rather than one generic breaker wrapping both fetch and solve.
  - *Rejected:* a single shared breaker. A shared breaker firing on data corruption looks identical in logs to a slow solver, which is exactly the ambiguity that costs a debugging afternoon.
  - *Trade-off:* more code and two taxonomies to maintain, bought against never conflating "the data was bad" with "the solver was slow."
  - **The R1.3-vs-R1.4c line (record it explicitly, a reviewer will ask):** R1.3 pre-flight validation answers *is this problem solvable*: structural and physical feasibility of the optimization inputs (SoC window, inverter cap, horizon). R1.4c answers *can this data be trusted*: provenance and integrity of the fetched series before it becomes an optimization input at all. R1.3 runs on a well-formed request; R1.4c runs on the wire. They do not overlap: a series can pass R1.4c (trustworthy) and still fail R1.3 (infeasible battery window), or fail R1.4c (stuck feed) while being structurally fine.

- **ADR-0013, one shared degradation vocabulary across both breakers.** The solver breaker reports `mode ∈ {optimal, fallback_greedy}` (R1.5). The ingestion breaker reports `status ∈ {healthy, outage, anomaly}`. Where a fetch feeds a solve (the backtest, a batch pipeline, a future auto-fetch path), the two compose into one honest provenance statement rather than two disconnected flags: a schedule computed on stale cached prices after an anomaly is **not** "healthy" just because the solver returned `optimal`.
  - *Rejected:* letting each breaker log independently with no shared status. That reproduces the silent-stale-dispatch hole, a solver `optimal` on stale data reads as fully healthy downstream.
  - *Trade-off:* a small shared value object (`GuardResult`) that lives in the `data` leaf and is read (never imported-upward) by consumers. `data` stays a leaf (import-linter); consumers attach the status, the guard never reaches up.
  - **Scope honesty:** R1.5's `POST /dispatch` takes client-supplied prices and does **not** fetch (R1.5 out-of-scope, unchanged). So R1.4c does not add fetch-inside-the-endpoint. The composition is exercised where a fetch actually feeds a solve (the backtest/example path), and the vocabulary is defined so a future auto-fetch endpoint attaches `data_status` next to `mode` for free.

## The anomaly checks (the real work; enumerate them, do not hand-wave)

The classifier's job is discrimination, and the sharp domain trap is here: **zero and negative prices are legitimate** in BE/NL day-ahead markets (high-renewable/low-demand windows; the repo already models a negative-price "paid load", golden oracle 4). A naive "flag €0.00 / flag negatives" check would misclassify real market conditions as corruption and fall back to stale prices; causing the exact bad dispatch it is meant to prevent, in the opposite direction. The checks therefore key on *feed pathology*, not on *price level*:

| # | Check | Fires on | Deliberately does **not** fire on |
|---|-------|----------|-----------------------------------|
| 1 | **Stuck feed** | ≥ `max_repeat` consecutive **bit-identical** values (a frozen/stale feed) | a real low/zero/negative price that still *varies* slightly period to period |
| 2 | **Out-of-band** | value `< band_lo` or `> band_hi` (default `[-600, 5000]` €/MWh) | any value inside the band, including legitimate negatives and zeros |
| 3 | **Non-finite** | any `NaN` / `±inf` in the series | – |
| 4 | **Structural** (surfaced from `validate_price_series`) | timestamp gap in the expected grid, duplicate timestamp, non-UTC / unsorted index | a regular gap-free series |
| 5 | **Empty / short** | zero points, or fewer than one expected day | a full 24/96-slot day |

Outage class (transport, before any content check): timeout, connection error, HTTP 5xx, or an empty HTTP body. Distinct from anomaly by construction, an outage means *no present data*; an anomaly means *present but untrustworthy data*.

`band_lo`, `band_hi`, and the stuck-feed threshold are configuration, not literals (see below). The band is **grounded in the EPEX SDAC harmonised clearing-price limits** (currently min −600 €/MWh from 2026-05-28, max 4000 €/MWh base, escalatable by +1000 steps during scarcity), with one escalation step of headroom → default `[-600, 5000]`. A value outside it *cannot be a real day-ahead clearing price*, so it is corruption by definition (a €99999 spike, a €−9000 sign error). This is a market technical bound, **not** the revenue-year-specific [sanity band](../formulation.md#sanity-band-gate-d); it does not shift with volatility, only if SDAC itself changes its limits.

## Parameters / configuration

| Item | Where | Default |
| --- | --- | --- |
| `sanity_band` `(lo, hi)` €/MWh | guard config (dataclass, env-overridable for ops) | `(-600.0, 5000.0)`: EPEX SDAC limits + headroom |
| `max_flat_hours` (stuck-feed threshold) | guard config | `8.0` wall-clock hours; `max_repeat = ceil(max_flat_hours / dt_hours)` → 8 at 60-min, 32 at 15-min |
| `expected_slots_per_day` | derived from series `dt` (`24` at 60-min, `96` at 15-min) | derived |
| last-known-good source | the existing `data/cache/*.parquet` (R1.4b), gitignored | per zone/window |

Operational knobs are guard config with env-overridable defaults ([conventions §5](../conventions.md#5-configuration)). The stuck-feed threshold is expressed in **wall-clock hours**, not slot count, so it survives the 60→15-min resolution switch; a stuck feed is frozen indefinitely, so 8 h catches it while never firing on a genuinely flat-but-real overnight period. Sanity-check the longest legitimate bit-identical run against a real slice in the token-gated integration test.

## Interfaces

```python
# src/bess/data/ingestion_guard.py  (data stays a leaf: imports nothing else in bess)
from enum import Enum

class FeedStatus(str, Enum):
    HEALTHY = "healthy"
    OUTAGE  = "outage"
    ANOMALY = "anomaly"

@dataclass(frozen=True)
class GuardResult:
    status: FeedStatus
    prices: pd.Series          # HEALTHY → the fetched series; degraded → last-known-good cache
    reason: str | None         # the specific check that fired: "stuck_feed" | "out_of_band" |
                               #   "non_finite" | "schema:<detail>" | "empty" | "http_5xx" | "timeout" | ...
    degraded: bool             # True when status != HEALTHY (a fallback series was substituted)

def classify_series(series: pd.Series, *, sanity_band, max_repeat, expected_slots_per_day) \
        -> tuple[FeedStatus, str | None]:
    """Pure content classifier over an already-fetched series. HEALTHY or ANOMALY only
    (transport/outage is decided before this runs). No I/O, the un-fakeable core.
    `max_repeat` is the resolved slot count (the config layer converts max_flat_hours/dt)."""

def guarded_fetch(
    fetch_fn,                  # a zero-arg callable performing the real fetch (wraps R1.4b fetch_day_ahead)
    *,
    last_known_good,           # callable | Series | None: the cache fallback
    sanity_band=(-600.0, 5000.0),
    max_flat_hours=8.0,        # → max_repeat = ceil(max_flat_hours / dt) from the series freq
    expected_slots_per_day=None,
) -> GuardResult:
    """Fetch → classify → (fallback + log) → GuardResult. Never raises on a bad feed:
    outage and anomaly both fall back to last-known-good and are logged with the reason.
    Raises only if a fallback is needed and none is available (a genuine hard stop)."""
```

- `classify_series` is pure and side-effect-free so property tests hammer it directly.
- `guarded_fetch` owns the try/except: transport exceptions → `OUTAGE`; `ValueError` from `validate_price_series` → `ANOMALY(reason="schema:…")`; otherwise run `classify_series` on the valid series for content anomalies. Logging distinguishes `outage` from `anomaly` and names the check.

## Golden oracles (exact expected classifications)

| # | input feed | expected `status` / `reason` | why this case |
|---|------------|------------------------------|---------------|
| 1 | clean synthetic day (`synthetic_day_ahead`, 1 day) | `HEALTHY` / `None` | the guard passes good data through untouched |
| 2 | day with a €0.00 **bit-identical** block ≥ `max_repeat` long | `ANOMALY` / `stuck_feed` | a frozen feed is caught; and note it is the *repetition*, not the zero, that fires |
| 3 | day with one interior timestamp slot removed | `ANOMALY` / `schema:gap` | structural fault surfaced from `validate_price_series` as a classification, not a crash |
| 4 | day with a single €9999 point (`> band_hi`) | `ANOMALY` / `out_of_band` | an implausible spike is caught by the wide band |
| 5 | **legitimate** low day: real-shaped negatives/zeros that still **vary** each hour | `HEALTHY` / `None` | the domain trap, a real solar-glut day must **not** be flagged (anti-false-positive) |
| 6 | `fetch_fn` raises a timeout | `OUTAGE` / `timeout`, `degraded=True`, `prices == last_known_good` | transport failure falls back and is labeled outage, not anomaly |

## Property tests (invariants that hold for any generated input)

- **No corrupted series ever passes as healthy:** for any injected fault (zeros-block ≥ `max_repeat`, gap, duplicate, out-of-band, NaN), `status != HEALTHY`.
- **No false positives:** any schema-valid, fault-free series with arbitrary in-band prices (including negatives and zeros that vary) → `status == HEALTHY`.
- **Outage ⊥ anomaly:** transport-shaped failures classify `OUTAGE`; content-shaped faults classify `ANOMALY`; the two never collide in the logged `status`.
- **Fallback safety:** whenever `status != HEALTHY` and a `last_known_good` exists, `GuardResult.prices` is the fallback (a schema-valid series), never the corrupted one; `degraded is True`.

## Out of scope (explicit; do not build here)

- **Fetch inside `POST /dispatch`**: R1.5's client-supplied-prices contract is unchanged; the guard wraps the offline/batch fetch path, not the endpoint.
- **Retry/backoff policy tuning, alerting transports (PagerDuty/Slack/email)**: the guard *classifies and logs*; wiring a real alert sink is ops config, not this phase.
- **Forecast-drift monitoring**: that is R2.1b (a different signal: model decay, not feed corruption).
- **Imputation / auto-repair of a bad series**: the guard falls back to last-known-good; it does not interpolate or "fix" corrupted data (silent repair is itself a nightmare failure mode).
- **A general data-quality framework / dbt-style test runner**: a single focused module; the dbt/software-defined-asset framing is one ADR sentence, not a dependency.

## Resolved decisions (reviewed 2026-07-01)

1. **Stuck-feed threshold**: **resolved:** express as **wall-clock hours** (`max_flat_hours = 8.0`), converted to a slot count per resolution (`ceil(hours/dt)`), not a fixed slot count. A stuck feed is frozen indefinitely, so 8 h catches the pathology without false-positiving on a real flat overnight run, and the hours framing survives the 15-min switch. Longest legitimate flat run to be confirmed against a real slice in the integration test.
2. **Sanity band**: **resolved: `(-600, 5000)`**, grounded in the **EPEX SDAC harmonised clearing-price limits** (min −600 €/MWh from 2026-05-28, max 4000 €/MWh base + one +1000 escalation step of headroom), verified against EPEX/ENTSO-E this session. A market technical bound, not the year-specific revenue band; revalidate only if SDAC changes its limits.
3. **`GuardResult` home**: **resolved: the `data` leaf**, a frozen dataclass, *read* (not imported upward) by consumers exactly as `api` reads `Schedule` from `optimizer`. No new import edge; the "data is a leaf" contract stays KEPT.
4. **Ship-gating**: **resolved: R1.4c gates the Release-1 ship.** The master plan's R1 definition-of-done already includes it (checklist, hero visual #3, commit line); shipping "done" then bolting on a promised visual is weaker than shipping once, complete. R1.5 remains a defensible standalone fallback if an immediate publish is ever forced.
