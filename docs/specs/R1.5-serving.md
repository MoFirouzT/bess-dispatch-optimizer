# Spec R1.5. Production wrapper + graceful-degradation circuit breaker

**Status:** Approved; frozen for implementation (open questions resolved below)
**Release:** R1  **Depends on:** R1.1–R1.4 (solver, pre-flight validation, greedy heuristic)

## Objective

A FastAPI + Pydantic service that wraps the existing `solve()` behind one HTTP endpoint and, when the solver exceeds a latency budget or fails on otherwise-valid input, **falls back to the greedy schedule** instead of erroring (the circuit breaker), all packaged in a Docker image. This completes Release 1: the deterministic engine becomes a deployable service that degrades gracefully under solver stress rather than failing a dispatch.

## Formulation reference

**n/a: no new math.** R1.5 is serving and operations over the existing model. The optimal schedule is the R1.1/R1.2 `solve()`; the fallback is the greedy baseline already specified in `docs/formulation.md` § "R1.4. Backtest semantics" and implemented as `greedy_window`. No constraint, objective term, or efficiency placement changes.

## Design decisions (to be recorded as ADRs)

- **ADR-0010; relocate the greedy heuristic to `bess.optimizer`.** `greedy_window` currently lives in `bess.backtest.baselines`, but `backtest` is an *offline* tool the serving chain must not depend on (import-linter). The circuit breaker needs greedy at serving time, so move `greedy_window` (+ its `_trajectory`/`_liquidate` helpers) to `bess.optimizer.heuristics`; `backtest` re-imports it from there. Both `api → optimizer` and `backtest → optimizer` are allowed edges, so no contract is widened. Golden values for greedy are unchanged (pure move).
- **ADR-0011; circuit-breaker semantics.** Two distinct failure classes, two distinct responses:
  - **Invalid input** (pre-flight `PreflightError`): the request is wrong, greedy cannot repair it. Return **HTTP 422** with the structured `ValidationIssue` list (code, field, message, context). No fallback.
  - **Valid input, solver does not deliver an optimum in budget** (HiGHS hits `time_limit`, returns non-optimal, or raises): return **HTTP 200** with the **greedy** schedule and `mode="fallback_greedy"`, and log the fallback. A dispatch is always served for valid input.

  The budget is enforced two ways at once: HiGHS `time_limit` (the solver stops itself) plus a wall-clock guard in the breaker (covers model build + load). Default budget **2.0 s**.

## Parameters / configuration

| Item | Where | Default |
| --- | --- | --- |
| `latency_budget_seconds` | app settings (Pydantic `BaseSettings`, env-overridable for ops) | `2.0` |
| `greedy_charge_pct` / `greedy_discharge_pct` | app settings | `20.0` / `80.0` (R1.4a) |
| battery parameters | request body (`BatterySpec`, Pydantic v2, reused as-is) | per request |

Operational knobs (budget, percentiles) are app settings with sane defaults, env-overridable; **model** parameters stay in the request `BatterySpec` (conventions §5). No secrets here (the ENTSO-E token belongs to the R1.4b data path, not serving).

## Interfaces

```
POST /dispatch
Request (application/json):
{
  "prices_eur_mwh": [float, ...],     # day-ahead curve, grid-side
  "dt_hours": 1.0,                    # > 0
  "battery": { ...BatterySpec... }    # capacity, soc_min, p_*_max, eta_*, soc_initial/terminal, ramp?, degradation?
}

Response 200 (application/json):
{
  "mode": "optimal" | "fallback_greedy",
  "objective_eur": float,
  "schedule": {
    "p_charge_mw":    [float, ...],   # length T, grid-side
    "p_discharge_mw": [float, ...],
    "soc_mwh":        [float, ...]
  },
  "solve_seconds": float,
  "solver_termination": "optimal" | "maxTimeLimit" | "fallback"
}

Response 422 (invalid input):
{ "issues": [ { "code": "...", "field": "...", "message": "...", "context": {...} }, ... ] }

GET /health -> 200 { "status": "ok", "solver": "appsi_highs", "solver_available": true }
```

The breaker is a thin service function `dispatch(request, *, budget, solve_fn=solve, greedy_fn=greedy_window)`: `solve_fn`/`greedy_fn` injectable so tests force each path deterministically without monkeypatching globals.

## Golden oracles (exact expected values)

| # | inputs | expected | why this case |
|---|--------|----------|---------------|
| 1 | `POST /dispatch`, R1.1 oracle 1 prices `[10,50,20]`, default 1 MWh/1 MW spec, `dt=1` | `mode="optimal"`, `objective_eur≈40.0`, schedule = R1.1 oracle 1 | the service returns exactly what `solve()` returns; pins the response schema to the engine |

## Property tests (invariants that hold for any valid input)

- **Every served schedule is physically feasible** regardless of `mode`: `0 ≤ p_charge ≤ P̄_ch`, `0 ≤ p_discharge ≤ P̄_dis`, no simultaneous charge+discharge, `soc ∈ [e_min, e_max]`, exact SoC continuity, ends at `soc_terminal`. (Reuse the R1.4 `_satisfies` invariant against the response.)
- **`mode="optimal"` objective ≥ `mode="fallback_greedy"` objective** on the same input when both are computed (the fallback is a floor, never beats the optimum; `V_greedy ≤ V*`, formulation §R1.4).
- **The breaker never raises on valid input:** for any pre-flight-valid request, `dispatch` returns a 200 with a feasible schedule, even when `solve_fn` raises.

## Out of scope (explicit; do not build here)

- **Kubernetes, compose stacks, autoscaling, dashboards/metrics UIs**: a single Dockerized service only (guardrail: no K8s, no dashboards).
- **AuthN/AuthZ, rate limiting, multi-tenancy**: not a public service in R1.
- **Async job queue / streaming**: the day-ahead solve is sub-second; one synchronous request/response.
- **Live ENTSO-E fetch inside the endpoint**: the client passes prices; the data path (R1.4b) stays separate.
- **Forecasting, scenarios, stochastic/recourse**: Release 2.
- **Warm-start / persistent solver process**: an R2.3 latency optimization, not needed at R1 horizons.

## Open questions: resolved at review (2026-06-26)

1. **Solver result surface**: **resolved:** add an optional `time_limit` param to `solve()` and a back-compatible `termination: str = "optimal"` field on `Schedule`. One return type; existing callers unaffected by the default.
2. **Budget default**: **resolved: 2.0 s.** Comfortable for a ≤96-period MILP; the breaker fires only on genuine solver stress, and tests force the fallback path via injection regardless of the budget.
3. **CI Docker build**: **resolved: build + `/health` smoke.** CI builds the image and curls `/health`; graceful degradation in deployment is the point of the phase.
