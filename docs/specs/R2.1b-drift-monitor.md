# Spec R2.1b — Rolling forecast-drift monitor

**Status:** Approved — frozen for implementation (design choices resolved inline)
**Release:** R2  **Depends on:** R2.1 (the forecaster whose output it watches), R1.4 (walk-forward evaluation)

## Objective

A rolling monitor over the R2.1 forecaster that flags when its accuracy degrades and, the actual skill, **classifies why**: a **regime shift** (the market genuinely changed; a naive baseline degrades too) versus **model staleness** (this model decayed specifically, while a naive baseline still tracks). It emits a typed `DriftReport`, not an undifferentiated "drift detected". This turns the forecaster from "accurate" into "accurate and self-aware", and is the slow-failure counterpart to R1.5b's fast-failure ingestion guard.

## Formulation reference

**n/a — engineering / monitoring phase, no new optimizer or forecast math.** PSI and a naive-relative error ratio are standard statistics, not a new modelling theory; the R2.1 conformal guarantee and the R1.4 walk-forward discipline still govern. No `formulation.md` delta (like R1.3 / R1.5b).

## The classification logic (the real work)

Two signals, and a **precedence** that is the whole design:

- **Relative error vs. a naive baseline** — `error_ratio = forecaster_MAE / naive_MAE` over the trailing window, where the naive baseline is a **seasonal-naive** forecast (`ŷ_t = π_{t−season}`, default `season = 168 h` so it captures weekday/weekend structure). A model that is much worse than "same hour last week" has *itself* decayed.
- **Population Stability Index (PSI)** on the recent input-price distribution vs. a reference window — high PSI means the *inputs* moved, i.e. the world changed.

**Precedence (ADR-0015):**

1. `error_ratio ≥ staleness_ratio` (default `1.3`) → **STALENESS.** Checked first: even under a regime shift, a healthy model should degrade *no worse than* the naive baseline; being materially worse than naive is model-specific decay → retrain.
2. else `psi ≥ psi_warn` (default `0.2`, the conventional "significant shift" threshold) → **REGIME_SHIFT.** Inputs moved but the model is still competitive with naive → recalibrate / accept, don't retrain.
3. else → **HEALTHY.**

Empirical interval **coverage** over the window is reported alongside (a coverage collapse corroborates drift) but attribution stays with `error_ratio` + `psi`, so the flag type is explainable.

## Honest gate framing

Unlike the MILP golden oracles (hand-solved arithmetic), this is a **behavioral regression gate**: the synthetic regime-shift and staleness patterns are *designed*, and the classifier is checked to separate them. It proves the classification *logic behaves as specified*, not a distribution-free truth. Stated plainly so the gate is not overclaimed as un-fakeable.

## Gates

- **Discrimination (headline, property test):** inject (a) a **regime shift** (a mean-level jump in the recent window so PSI is high, with forecaster and naive errors rising *together*, ratio ≈ 1) → `REGIME_SHIFT`; and (b) **staleness** (inputs stable, PSI low, but forecaster residuals inflated while naive tracks, ratio high) → `STALENESS`. The monitor must classify the two **differently and correctly**.
- **Classifier unit cases:** `classify_drift` returns STALENESS / REGIME_SHIFT / HEALTHY for representative metric triples, and the precedence holds when both signals fire (staleness wins).
- **PSI sanity:** `psi(x, x) ≈ 0`; `psi` grows monotonically as a distribution is shifted further from its reference.

## Out of scope (explicit — do not build here)

- **Automatic retraining / recalibration triggers** — the monitor *classifies and logs*; wiring it to fire a retrain is a follow-on, not this phase (silent auto-retrain is itself a failure mode).
- **Adaptive conformal (ACI) for distribution shift** — noted in `formulation.md` §R2.1 as future; the monitor detects, it does not adapt the intervals online.
- **Multivariate / feature-level drift attribution** — PSI on the price distribution only, not per-feature.
- **Alerting transports** (PagerDuty/Slack) — a log line; wiring a sink is ops config.

## Open questions — resolved (2026-07-01)

1. **Naive baseline** — resolved: **seasonal-naive at 168 h** (weekly), the honest strong benchmark for hourly power prices (weekday/weekend structure); configurable.
2. **Precedence when both signals fire** — resolved: **staleness wins** (ADR-0015). Worse-than-naive is model-specific decay regardless of input shift.
3. **Thresholds** — resolved: `psi_warn = 0.2` (standard), `staleness_ratio = 1.3`; both configurable, to be sanity-checked against real ENTSO-E drift episodes if/when available.
