# Spec R2.1b. Rolling forecast-drift monitor

**Status:** Approved; frozen for implementation (design choices resolved inline)
**Release:** R2  **Depends on:** R2.1 (the forecaster whose output it watches), R1.4 (walk-forward evaluation)

## Objective

A rolling monitor over the R2.1 forecaster that flags when its accuracy degrades and, the actual skill, **classifies why**: a **regime shift** (the market genuinely changed; a naive baseline degrades too) versus **model staleness** (this model decayed specifically, while a naive baseline still tracks). It emits a typed `DriftReport`, not an undifferentiated "drift detected". This turns the forecaster from "accurate" into "accurate and self-aware", and is the slow-failure counterpart to R1.4c's fast-failure ingestion guard.

## Formulation reference

**n/a; engineering / monitoring phase, no new optimizer or forecast math.** PSI and a naive-relative error ratio are standard statistics, not a new modelling theory; the R2.1 conformal guarantee and the R1.4 walk-forward discipline still govern. No `formulation.md` delta (like R1.3 / R1.4c).

## The classification logic (the real work)

Two signals, and a **precedence** that is the whole design:

- **Relative error vs. a naive baseline**: `error_ratio = forecaster_MAE / naive_MAE` over the trailing window, where the naive baseline is a **seasonal-naive** forecast (`ŷ_t = π_{t−season}`, default `season = 168 h` so it captures weekday/weekend structure). A model that is much worse than "same hour last week" has *itself* decayed.
- **Population Stability Index (PSI)** on the recent input-price distribution vs. a reference window; high PSI means the *inputs* moved, i.e. the world changed.

**Precedence (ADR-0015):**

1. `error_ratio ≥ staleness_ratio` (default `1.3`) → **STALENESS.** Checked first: even under a regime shift, a healthy model should degrade *no worse than* the naive baseline; being materially worse than naive is model-specific decay → retrain.
2. else `psi ≥ psi_warn` (default `0.2`, the conventional "significant shift" threshold) → **REGIME_SHIFT.** Inputs moved but the model is still competitive with naive → accept / adapt, don't retrain.
3. else `coverage` present and `coverage ≤ confidence_level − coverage_tol` (default `0.10`) with at least `min_coverage_samples` (default `100`) points → **MISCALIBRATION.** Point model tracks and inputs are stable, but the intervals under-cover → recalibrate the conformal layer. Checked after regime so a regime shift (which also breaks coverage) keeps its attribution.
4. else → **HEALTHY.**

Precedence extension recorded in **ADR-0016** (amends ADR-0015). Coverage is one-sided: over-coverage (intervals too wide) is reported as a signed deviation but never alarms.

Empirical interval **coverage** over the window is a **third, first-class signal** (amendment 2026-07-03). The forecaster's product is *calibrated intervals*; coverage decalibration is orthogonal to both point-accuracy decay (staleness) and input drift (regime), a model can hold `error_ratio ≈ 1` and `psi` low while its 90% band silently covers 75%. Since R2.2 samples scenarios from these intervals, an unmonitored coverage collapse propagates straight into the stochastic layer. It is checked **after** regime so a genuine regime shift keeps the "market moved" attribution, and it is one-sided (only *under*-coverage alarms; over-wide intervals are reported, not flagged):

- **Empirical coverage vs. nominal**: `coverage = mean(lower ≤ realized ≤ upper)` over the window; a breach is `coverage ≤ confidence_level − coverage_tol`. Maps to a distinct remedy: **recalibrate** the conformal layer (the forecaster's `recalibrate()`), not retrain.

## Honest gate framing

Unlike the MILP golden oracles (hand-solved arithmetic), this is a **behavioral regression gate**: the synthetic regime-shift and staleness patterns are *designed*, and the classifier is checked to separate them. It proves the classification *logic behaves as specified*, not a distribution-free truth. Stated plainly so the gate is not overclaimed as un-fakeable.

## Gates

- **Discrimination (headline, property test):** inject (a) a **regime shift** (a mean-level jump in the recent window so PSI is high, with forecaster and naive errors rising *together*, ratio ≈ 1) → `REGIME_SHIFT`; and (b) **staleness** (inputs stable, PSI low, but forecaster residuals inflated while naive tracks, ratio high) → `STALENESS`. The monitor must classify the two **differently and correctly**.
- **Miscalibration discrimination (headline for this amendment, property test):** inject an episode where the **point model tracks** (`error_ratio ≈ 1`) and **inputs are stable** (PSI low) but the intervals are too tight so realized prices fall outside the band far more than `1 − confidence_level` of the time → `MISCALIBRATION`, *not* HEALTHY and *not* STALENESS/REGIME. The mirror case (same point/PSI, correctly wide intervals) → `HEALTHY`. Proves coverage is a live signal, not a passenger.
- **Classifier unit cases:** `classify_drift` returns STALENESS / REGIME_SHIFT / MISCALIBRATION / HEALTHY for representative metric tuples, and the precedence holds when signals co-fire (staleness > regime > miscalibration); a coverage breach below `min_coverage_samples` stays HEALTHY (informational only); over-coverage never flags.
- **PSI sanity:** `psi(x, x) ≈ 0`; `psi` grows monotonically as a distribution is shifted further from its reference.

## Out of scope (explicit; do not build here)

- **Automatic retraining / recalibration triggers**: the monitor *classifies and logs*; wiring it to fire a retrain is a follow-on, not this phase (silent auto-retrain is itself a failure mode).
- **Adaptive conformal (ACI) for distribution shift**: noted in `formulation.md` §R2.1 as future; the monitor detects, it does not adapt the intervals online.
- **Multivariate / feature-level drift attribution**: PSI on the price distribution only, not per-feature.
- **Alerting transports** (PagerDuty/Slack), a log line; wiring a sink is ops config.

## Amendment: coverage as a first-class drift signal (2026-07-03)

**Motivation.** The forecaster's product is *calibrated intervals*, but the original monitor watched only point-MAE (staleness) and input-PSI (regime); coverage was computed and carried in `DriftReport` yet never drove classification. Interval decalibration is an orthogonal failure mode (stable point accuracy + stable inputs, but under-covering bands) that the two original signals miss, and it propagates directly into R2.2's interval-sampled scenarios.

**Change.** Add `DriftStatus.MISCALIBRATION`, checked after regime (§ Precedence step 3), one-sided (only under-coverage alarms), guarded by `min_coverage_samples` for small-window noise. Maps to a distinct remedy; recalibrate the conformal layer, not retrain. New headline property gate (§ Gates). No `formulation.md` delta (still monitoring). New **ADR-0016** amends ADR-0015's precedence.

**Open decisions (recommendations shown, override at will):**
1. **Precedence**: recommend staleness > regime > miscalibration (keeps ADR-0015 intact; regime shift retains attribution when coverage also drops). Alternative: miscalibration before regime if you want "recalibrate" surfaced even under input shift.
2. **Threshold / direction**: recommend `coverage_tol = 0.10` one-sided with `min_coverage_samples = 100`. Alternative: a one-sided binomial test on the miss count (more principled on small windows but more machinery, likely over-built for this phase).
