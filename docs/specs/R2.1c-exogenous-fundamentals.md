# Spec R2.1c. Exogenous day-ahead fundamentals features (residual load)

**Status:** Implemented (gate green; five resolved decisions below, [ADR-0024](../decisions/0024-day-ahead-forecast-feature-alignment.md))
**Release:** R2  **Depends on:** R2.1 (forecaster + leakage-safe feature matrix), R1.4b (ENTSO-E loader to mirror for the new series), R1.4c (ingestion guard: the fundamentals fetch is guarded like the price fetch), R1.4a (walk-forward + gate-closure information set)

## Objective

Extend the R2.1 forecaster's feature set with **exogenous day-ahead fundamentals** fetched from ENTSO-E for the same bidding zone: the day-ahead **load forecast** and the day-ahead **wind + solar generation forecast**, combined into **residual load** (`load − wind − solar`), the dominant driver of the day-ahead price. This realizes the exogenous-feature slot R2.1 deferred, conditions the price forecast on *why* prices move rather than only their own past, and does so without a new data provider or a change to the optimizer.

## Motivation (the gap this closes)

The R2.1 feature set is **purely autoregressive + calendar** (price lags + hour/weekday/month/holiday). A day-ahead price is the clearing point of an auction whose position on the supply (merit-order) stack is set by **residual load**: high load with low renewables pushes clearing up the stack toward the expensive, volatile marginal units; abundant renewables pushes it down toward zero or negative. A price-lag model can only learn this second-hand, through yesterday's prices. Feeding residual load injects the driver directly. Two consequences, both measured by the gate below:

- **Fundamentals (this phase).** Conditioning on the day-ahead residual-load forecast should improve point/quantile accuracy over the price+calendar model, and must not degrade interval coverage.
- **Merit-order structure, for free (why no structural model is needed).** LightGBM is a tree ensemble and natively learns step/threshold functions, so given residual load it approximates the supply curve's kinks empirically. This is the cheap, reduced-form way to capture the structural non-linearity of the auction without building a plant-level supply stack (out of scope; see below). A stronger explicit stack feature is posed as an open question, not built by default.

This changes **nothing** about the price-taker assumption ([`formulation.md` §R1.1](../formulation.md#modeling-notes), "Price-taker" note): the forecaster still treats price as an exogenous series to predict; it simply conditions that prediction on more of the exogenous state.

## Formulation reference

**No new math, no optimizer delta.** R2.1c adds features to an existing learner; it introduces no constraint, objective term, efficiency placement, or new coverage theory. `formulation.md`'s §R2.1 conformal summary and the dispatch math are unchanged. Residual load as a merit-order proxy is domain context, not a formulation claim, so no `formulation.md` section is added (consistent with R2.1b / R1.4c, which are pure engineering phases with no section).

## Governing reference

None required (CLAUDE.md §1, standard feature engineering; the merit-order intuition is textbook-ubiquitous domain context, not a new method). Pointers only, no reliance:

- **Domain context (residual load / merit order):** the day-ahead price as the intersection of a step supply stack and demand; residual load = load − must-run renewables as the stack position. Recorded in `references.md` as context, verify-before-citing if any specific figure is quoted.
- **Exogenous regressors in forecasting (pointer):** Hyndman & Athanasopoulos, *Forecasting: Principles and Practice*, on including external predictors and honest out-of-sample evaluation.

## The leakage subtlety (correctness-critical)

R2.1's price features are lagged `≥ 24 h` precisely because *realized* prices for day `D` are not known until the day-ahead auction clears. **Fundamentals are different, and the difference is the whole point:** ENTSO-E publishes the load forecast and the wind/solar generation forecast for day `D` **before** the day-ahead gate closes on `D−1`. So for a target timestamp `t` on day `D`, the day-ahead *forecast* of load/renewables **at `t`** is already in the gate-closure information set. These features are therefore aligned **contemporaneously to the target `t`** (not shifted into the past like price lags).

This is leakage-safe **only because the feature is the published day-ahead forecast, never the realized actual.** Using `load_actual(t)` or `wind_actual(t)` would be look-ahead and is forbidden. The leakage gate below pins this: the feature at `t` must be invariant to any mutation of the *realized* series at `t`, and must equal the *forecast* series at `t`. This is the R2.1c analogue of R2.1's no-leakage invariant, and the one place this phase can go silently wrong.

## Scope decisions

- **Source: ENTSO-E only, day-ahead-published series only.** Reuses the R1.4b client, token, cache, and R1.4c ingestion guard. This is the boundary that keeps the phase cheap: one provider, one timing convention, no new licensing path.
- **Fundamentals, not raw weather.** Wind/solar *generation forecasts* already fold hub-height wind and irradiance into the quantity that actually sets the stack (MW of must-run renewables), and share ENTSO-E's provider and day-ahead timing. Raw weather (temperature, wind speed) would be a separate provider + licensing path (the reason R2.1 deferred it) for a driver that generation forecasts already summarize. Residual load supersedes the deferred "weather as R2.1c" idea.
- **Gas / carbon / cross-border flows are out.** Real drivers, but different providers (commodity markets) and messier timing; deferred to keep scope contained. Residual load is the single highest-value feature per unit of plumbing.
- **Graceful degradation.** A fundamentals series can be missing or partial for a window. When a required exogenous series is unavailable, the forecaster falls back to the R2.1 price+calendar feature set (logged), rather than failing the forecast. This mirrors the R1.5 breaker / R1.4c reliability posture.

## Golden oracles

Feature-construction is exact arithmetic (no MILP here), so these are un-fakeable like R2.1's leakage checks, not statistical.

| # | inputs | expected feature output | why this case |
|---|--------|-------------------------|---------------|
| 1 | fundamentals `load_da=[100,120]`, `wind_da=[30,10]`, `solar_da=[20,0]` at targets `t0,t1` | `residual_load = [50, 110]` at `t0,t1` | pins the residual-load arithmetic exactly |
| 2 | same, but mutate the *realized* load at `t1` to any value | `residual_load` feature at `t1` unchanged | pins leakage-safety: features read the forecast series, never the actual |
| 3 | `fundamentals=None` | feature matrix byte-identical to R2.1 `make_features` | pins the opt-in contract: off ⇒ exactly R2.1 |
| 4 | fundamentals forecast at `t` aligned to target `t` | the `residual_load` at row `t` equals the forecast for `t`, not for `t−1` | pins contemporaneous (not lagged) alignment |

## Property tests

- **Leakage invariance:** the fundamentals features at `t` are invariant to any mutation of the *realized* load/wind/solar at `t` or later, and depend only on the published day-ahead forecast series (the un-fakeable check, R2.1c analogue of R2.1's no-leakage property).
- **Opt-in identity:** with `use_fundamentals=False` (or `fundamentals=None`), outputs are bit-identical to the R2.1 forecaster on the same seed and inputs.
- **Residual-load identity:** `residual_load == load_da − wind_da − solar_da` row-wise, for all synthetic draws.
- **No new NaNs:** adding fundamentals introduces no NaN rows beyond R2.1's lag warm-up (alignment is clean or the row is dropped consistently).
- **Determinism:** fixed seed + fixed inputs (prices + fundamentals) → bit-stable point/interval outputs.
- **Graceful degradation:** with fundamentals absent, the forecaster produces a valid R2.1-equivalent forecast and logs the fallback.

## Out of scope

- **EVT / spike tail on scenarios (the #2 tail work).** Conditioning the forecast on residual load is the *prerequisite* for a conditional spike model (spikes are residual-load-extreme events), but fitting a Generalized Pareto tail to the residual bootstrap lives in the **scenario layer** (a future R2.2 addendum), not the forecaster. Deferred deliberately; noted so the dependency ordering (fundamentals → conditional tail) is on record.
- **Explicit plant-level merit-order / supply-stack model.** Fuel prices, unit marginal costs, capacities, outages: a different data-acquisition and modeling scope. The tree + residual-load reduced form is the intended substitute.
- **Gas / carbon / cross-border flow features.** Different providers/timing; deferred.
- **Raw weather features.** Superseded by generation forecasts (which encode weather) for the price-forecasting purpose.
- **Intraday / imbalance markets.** R3 scope.
- **Any optimizer or formulation change.** This is a forecaster feature phase.

## Open questions

Phase-local decisions only; posed with a proposed answer, resolved in place at review (2026-07-24). Resolved lines kept as the decision trail.

- **Which series enter the matrix: residual load only, or the components too?** *Proposed:* fetch load + wind + solar day-ahead forecasts, feed `residual_load` as the primary engineered feature plus the three raw components, and let the tree weight them. **Resolved (2026-07-24): both.** `residual_load` carries the merit-order signal; the three raw components let the model split economically distinct regimes (solar-driven midday troughs vs. wind-driven overnight ones) that share a residual-load value. Cheap once the series are fetched; the property test pins `residual_load == load_da − wind_da − solar_da` regardless.
- **Contemporaneous alignment of the forecast features to the target `t` (the leakage subtlety).** *Proposed:* align day-ahead forecast features to `t` itself (they are published pre-gate-closure), not lagged like price. **Resolved (2026-07-24): yes, contemporaneous alignment, promoted to [ADR-0024](../decisions/0024-day-ahead-forecast-feature-alignment.md)** (to be written at implementation, following the R2.3 precedent of naming ADRs in-spec and authoring them on green). The ADR records the rule: *day-ahead forecast features are known at gate closure, so aligning them to the target `t` is leakage-safe if and only if the series is the published forecast, never the realized actual.* Pinned by golden oracles 2 and 4 and the leakage-invariance property. This is the single correctness-critical choice in the phase.
- **An explicit reduced-form stack feature (the #1 "stack transform")?** A walk-forward empirical `price ≈ f(residual_load)` curve (binned median), fed as a prior the tree corrects. **Resolved (2026-07-24): defer.** Ship residual load first; trees already learn the stack from it. Add the transform only if the fitted partial-dependence of price on residual load visibly under-fits the kink on real data. Keeps this phase minimal; recorded as named future work, not built here.
- **Missing-fundamentals policy.** *Proposed:* fall back to the R2.1 price+calendar feature set for windows where a required series is unavailable, logged, rather than failing. **Resolved (2026-07-24): graceful fallback**, matching the R1.5 breaker / R1.4c ingestion-guard reliability posture (a degraded-but-valid forecast beats no forecast). The fallback is logged at `WARNING` and covered by the graceful-degradation property test.
- **Phase id / naming.** *Proposed:* R2.1c, realizing the exogenous-feature slot R2.1's spec named ("weather as R2.1c or named future work"). **Resolved (2026-07-24): R2.1c**, with fundamentals (load + wind/solar forecasts) chosen over raw weather because generation forecasts already encode weather into the quantity that sets the stack and share ENTSO-E's provider and day-ahead timing.
