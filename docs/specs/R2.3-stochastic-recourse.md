## Objective

Make the stochastic layer *earn its place*: a risk-aware two-stage dispatch program over R2.2's scenario set, plus an intraday receding-horizon (MPC) recourse mechanism, that together produce **a measured value-of-stochastic-solution (VSS) > 0** and a **risk-vs-return frontier** where a CVaR-averse solution demonstrably reduces downside versus the risk-neutral one under price-forecast error, both out-of-sample. This is the first Release-2 phase with a real optimizer delta (`formulation.md` §R2.3): it adds recourse variables, a CVaR risk term, and a non-anticipativity structure to the R1 model.

## The design driver: the VSS-collapse trap

A naive day-ahead two-stage stochastic program for this battery collapses to the mean-value problem, giving **VSS = 0**. The reason is structural, and stating it precisely is the point of the phase: the objective is *linear* in the price and the feasible set (SoC dynamics, power limits, ramp) does *not* depend on the price realization. If the entire 24-hour dispatch is a single here-and-now decision $x$ committed before prices are known, the stochastic objective is

$$\max_x \ \mathbb{E}_s\big[\pi^{(s)}\cdot x\big] \;=\; \big(\mathbb{E}_s[\pi^{(s)}]\big)\cdot x \;=\; \bar\pi\cdot x,$$

exactly the mean-value (EV) problem, so the stochastic solution equals the EV solution and VSS = 0. A layer that adds scenarios but returns the mean-value schedule is *hollow*; the gate exists to forbid that outcome.

Value comes from breaking one of the two collapse conditions. R2.3 breaks both, deliberately:

1. **Genuine recourse against a distinct realized signal (the two-stage escape).** The here-and-now decision is a *day-ahead commitment*; the recourse decision is an *intraday re-dispatch against realized prices* that were uncertain at gate closure, with SoC continuity linking the two stages. Because the second-stage (recourse) value function $Q(x,\pi)$ is the value of an LP parameterised by the realized price, it is **convex in $\pi$**; by Jensen $\mathbb{E}_\pi[Q(x,\pi)] \ge Q(x,\bar\pi)$, and the first-stage decision that maximises the expectation differs from the one that maximises the mean-case value. That difference is the VSS. (Open question 1 fixes the exact construction; the recommendation is a day-ahead / intraday two-price structure so the recourse signal is genuinely distinct from the commitment signal.)
2. **A risk-averse objective (value even single-shot).** A CVaR term is piecewise-linear, not linear, in the scenario outcomes, so the expectation-collapse does not apply: risk aversion bends the objective and moves the decision even without recourse. This is what the risk-vs-return frontier traces.

## Formulation reference

Implements [`formulation.md` §R2.3](../formulation.md#r23-risk-aware-two-stage-dispatch--intraday-recourse-optimizer-delta), summarized here in house notation (grid-side power, per-unit SoC config, `π/e/η/Δt`; the R1.1 model is the second-stage physics, unchanged):

**Scenario input (from R2.2).** A reduced set $\{(\pi^{(s)}, p_s)\}_{s=1}^{S}$, $p_s \ge 0$, $\sum_s p_s = 1$, each $\pi^{(s)}$ a full-horizon price path (€/MWh, grid-side, UTC).

**Two-stage variables.** First-stage (here-and-now, shared across all scenarios; non-anticipative) commitment $x$; second-stage (recourse, per scenario $s$) dispatch variables $\{p^{ch,(s)}_t, p^{dis,(s)}_t, e^{(s)}_t, u^{(s)}_t\}$ obeying the full R1.1 physics (SoC balance, bounds, mutual-exclusion binary, ramp, terminal SoC) *for each scenario*, coupled to $x$ by the stage-linking constraint fixed in open question 1.

**Risk-aware objective (CVaR mean-risk; Rockafellar-Uryasev linearisation).** Let $\text{profit}_s(x)$ be the realised revenue in scenario $s$ (first-stage cash flow + second-stage recourse cash flow, grid-side, no efficiency term). With loss $L_s = -\text{profit}_s$, tail level $\alpha$ (e.g. 0.95), and risk weight $\lambda \in [0,1]$:

$$\max_{x,\,\eta,\,z_s \ge 0} \ (1-\lambda)\sum_s p_s\,\text{profit}_s(x) \;-\; \lambda\Big(\eta + \tfrac{1}{1-\alpha}\sum_s p_s\, z_s\Big), \qquad z_s \ge L_s - \eta.$$

The bracket is $\text{CVaR}_\alpha$ of the loss (Rockafellar-Uryasev); $\eta$ recovers the Value-at-Risk at optimum. $\lambda = 0$ is the risk-neutral expectation; sweeping $\lambda$ traces the **mean-CVaR frontier**. The term is LP-representable, so the program stays a (MI)LP solvable by HiGHS; the only integrality is the per-scenario mutual-exclusion binaries.

**Recourse / MPC (receding horizon).** The intraday mechanism executes the first-stage commitment, then re-solves the *remaining* horizon at each step as prices realise, carrying SoC as the linking state. Structurally model-predictive control: plant model = SoC dynamics (R1.1 constraint (1)), state continuity across windows, a re-optimisation trigger per period, price forecast as the disturbance. It reuses the R1.1 deterministic `solve()` per window (`dt` per-solve already supports a 15-/60-min granularity change intraday).

**Decision-value metrics (Birge-Louveaux), tying to R1.4's ceiling.** Define, over the scenario set / out-of-sample realisations:
- **EV** = mean-value problem (solve once at $\bar\pi$).
- **RP** = recourse problem value = optimal stochastic objective (risk-neutral for the VSS definition, $\lambda=0$).
- **EEV** = expected result of using the EV solution: fix the EV first-stage $\bar x$, evaluate its expected value with optimal recourse.
- **WS** = wait-and-see = $\sum_s p_s\,V^\star(\pi^{(s)})$, the scenario-averaged *perfect-foresight* value; this is exactly R1.4's ceiling averaged over scenarios.
- **VSS** $= \text{RP} - \text{EEV} \ge 0$;  **EVPI** $= \text{WS} - \text{RP} \ge 0$.

Ordering (a correctness gate, extending R1.4's $V^{\text{greedy}} \le V^{\text{roll}} \le V^\star$): $\text{EEV} \le \text{RP} \le \text{WS}$.

**Duration axis (ADR-0022).** VSS scales with storage duration: near-zero headroom for a 1-hour asset (deterministic day-ahead is near-solved), real headroom for a 4-hour asset. A single-duration VSS therefore understates or overstates the layer's general value, so any reported VSS names its duration, and a VSS-vs-duration sweep ({1h, 2h, 4h}) is the honest headline once R1.4's duration sweep lands. This is a reporting requirement on the existing metrics, not a model change; see [ADR-0022](../decisions/0022-storage-duration-reported-axis.md).

**Considered but out of scope (recorded in the delta):** Bertsimas-Sim $\Gamma$-budget robust counterpart (an alternative to CVaR, noted not built; open question 2); multistage (>2) scenario trees; Benders / L-shaped decomposition (R2.4 / optional Julia); an explicit intraday order-book or imbalance-settlement market model (R3 scope; the recourse here re-trades against a realised price, it does not model market microstructure); chance constraints as hard probabilistic constraints (the soft CVaR objective is used instead; open question 2).

## Design decisions

- **ADR-0019. The two-stage construction that escapes VSS = 0.** Records the chosen here-and-now / recourse split (open question 1) and *why* it yields a strictly positive VSS where the naive single-signal day-ahead program does not. This is the phase's central decision; the ADR carries the trap analysis so it is not re-litigated.
- **ADR-0020. Risk model = CVaR mean-risk frontier, not $\Gamma$-budget robust.** CVaR is scenario-native (composes directly with R2.2's discrete set), coherent, and LP-representable; the Bertsimas-Sim robust counterpart is the compared/optional alternative (mirrors R2.1's CQR-vs-split and R2.2's forward-vs-kmeans baseline pattern). Open question 2.
- **ADR-0021. Recourse = receding-horizon MPC simulator; VSS estimated out-of-sample.** Records the MPC framing (warm-start, SoC continuity, granularity handling) and the estimation protocol (EEV/RP/WS on held-out realised paths, not in-sample on the scenarios the decision was fit to).

## Golden oracles

Unlike R2.1's statistical gates, these are **exact and hand-computable**: the collapse trap and the CVaR linearisation both have closed forms, so a stochastic phase still gets real oracles rather than only a distributional check.

| # | inputs | expected | why this case |
|---|--------|----------|---------------|
| 1 | a single-signal day-ahead set (whole schedule first-stage, linear objective), any scenarios | **VSS = 0** exactly | pins that we reproduce the collapse trap, not paper over it: the naive construction *must* give VSS 0 |
| 2 | a hand-built value-generating instance (open-question-1 construction, 2–3 scenarios) | **VSS > 0** at a hand-computed value | pins the escape: the chosen recourse structure yields the exact positive VSS arithmetic predicts |
| 3 | 2-scenario CVaR case, known profits, fixed `alpha`, `lambda` | the exact CVaR, VaR ($\eta$), and mean-risk objective by hand | pins the Rockafellar-Uryasev linearisation against arithmetic |
| 4 | `lambda = 0` on any set | `solve_stochastic` first-stage == the risk-neutral RP == the EV schedule when the set collapses | pins that the risk term switches off cleanly and reduces to the expectation |

(Exact numbers filled in during test-writing, computed by hand, then locked at tolerance `1e-6`.)

## Property tests

- **VSS non-negativity:** `VSS = RP - EEV >= 0` and `EVPI = WS - RP >= 0` for every generated scenario set (the theory guarantees it; a violation is a formulation bug).
- **Value ordering:** `EEV <= RP <= WS` (extends R1.4's ordering to the stochastic setting).
- **Second-stage feasibility:** every scenario's recourse dispatch satisfies all R1.1 constraints (SoC balance/bounds, mutual exclusion, ramp, terminal) at its realised price.
- **Non-anticipativity:** the first-stage commitment is identical across scenarios (shared, not scenario-indexed).
- **CVaR monotonicity of the frontier:** as `lambda` increases, expected profit is non-increasing and CVaR (downside) is non-worsening; the frontier is monotone (no dominated points).
- **Risk reduces downside:** under a ±`price_error` stress, the CVaR-averse solution's realised worst-case / tail loss is `<=` the risk-neutral solution's (the risk term does real work).
- **Recourse never worse than commit-and-hold:** the MPC policy's realised value on a path is `>=` executing the fixed mean-forecast schedule (recourse can only help; equals R1.4's rolling logic at the deterministic limit).
- **Determinism:** fixed seed + inputs yield bit-stable schedules and metrics.

## Resolved decisions (reviewed 2026-07-09)

Phase-local decisions only; roadmap / positioning stays in the Tier 0 log. Each was posed with a proposed answer, resolved in place; the resolved lines are the decision trail.

1. **The two-stage construction that guarantees VSS > 0.** **Resolved: day-ahead-commitment + intraday-recourse two-price structure.** First-stage = day-ahead schedule at the (known) day-ahead price; second-stage = intraday re-dispatch at the *realised* price (the R2.2 scenario), SoC continuity linking the stages, the intraday adjustment the recourse. To stay self-contained and token-free-testable, the intraday/realised signal is derived from the same R2.2 scenario set (no new data feed); the day-ahead price is its scenario mean or a held-out realisation. Convex recourse value function ⇒ a clean, positive, hand-verifiable VSS; the single-signal receding-horizon variant is kept only as golden oracle 1 (the collapse). Shapes [ADR-0019]. *(The phase's load-bearing decision.)*
2. **Risk model: CVaR vs. Bertsimas-Sim $\Gamma$-robust vs. hard chance constraint.** **Resolved: CVaR mean-risk frontier primary** (scenario-native, coherent, LP-representable, composes with R2.2); the $\Gamma$-budget robust counterpart is the documented compared alternative, hard chance constraints out of scope. CVaR reuses the discrete scenario set directly and yields a smooth frontier ([ADR-0020]; mirrors ADR-0014 CQR-vs-split and ADR-0018 forward-vs-kmeans).
3. **One governing reference, or a primary-plus-authoritative-subordinates split?** **Resolved: keep R2.3 as one phase.** Name **Birge & Louveaux governing** (the two-stage/VSS spine); treat Shapiro-Dentcheva-Ruszczyński (CVaR) and Rawlings-Mayne-Diehl (MPC) as **subordinate-but-authoritative for their sub-concepts**, reconciled to house notation, exactly as R2.2 named a paper-pair governing with a textbook subordinate. The R2.3a/R2.3b split was considered and declined (a two-phase gate for one coherent value-generating layer is not worth the seam).
4. **VSS estimation protocol.** **Resolved: out-of-sample.** Estimate EEV/RP/WS on held-out realised paths (fit the decision on a reduced scenario set, evaluate on disjoint realisations under R1.4 walk-forward/leakage discipline), so the reported VSS is an honest generalisation number, not an in-sample artefact of the scenarios the decision saw ([ADR-0021]).
5. **New dependency?** **Resolved: none.** The CVaR program is LP-representable and solves with the existing `appsi_highs`; scenarios come from R2.2; metrics and the MPC loop are numpy over repeated `solve()` calls. No new solver, no `scipy`.
