# Spec R2.5b. Dispatch value of the scenario tail (value measurement)

**Status:** Implemented (gate green; four resolved decisions below, one corrected during implementation; the result is a measured null, which is the finding)
**Release:** R2  **Depends on:** R2.5 (the value-study machinery + out-of-sample protocol this reuses), R2.2b / R2.2c (the tail being evaluated), R2.2 (the plain-bootstrap baseline), R2.3 (the two-stage program), R2.1c (the residual-load covariate for the conditional tail)

## Objective

Measure, in **realized euros over real windows**, whether the extreme-value (R2.2b) / residual-load-conditional (R2.2c) scenario tail actually changes the two-stage dispatch **commitment and its realized value** versus the plain residual bootstrap (R2.2). Report the per-window **tail value (TV)** as a distribution, not a single number, and **not asserted positive**, exactly mirroring the R2.5 forecast-value baseline. No optimizer change.

## Motivation (closing the loop the tail work left open)

R2.2b un-capped the scenario spikes and R2.2c tied them to residual load, and both made the scenario *representation* measurably more faithful. But the README says plainly that the VSS and forecast-value numbers still use the **plain** bootstrap, so the tail's effect on the actual dispatch decision is **unmeasured**. There is a direct warning from this same project: the R2.5 forecast-value baseline found that a demonstrably better *forecast* did **not** convert to dispatch euros, because the intraday recourse budget already hedges day-shape error. The identical question applies to a better *tail*: does representing an unprecedented or tight-hour spike change what the battery commits to and reserve, or does the recourse already capture it after the fact?

This phase answers that. Its result is decision-useful either way, and it **gates R2.2d** (the deferred frequency-channel tail): if the tail moves the dispatch-value needle, a richer tail is justified; if TV is another null, the honest call is to stop refining the generator.

## The mechanism it probes

A day-ahead spike scenario makes the two-stage program **reserve stored energy** (or leave recourse room) so it can discharge into a realized spike. The plain bootstrap, blind to spikes beyond history, does not reserve. But the recourse budget `ρ` lets the program re-dispatch into a realized spike *after* the fact, within budget, so the tail's incremental value is only the part the day-ahead commitment captures that recourse cannot. Whether that is positive, zero, or (from over-reserving on spikes that never come) slightly negative is the empirical question. TV is therefore expected to depend on `ρ` (tight budget → commitment matters → tail can earn; generous budget → recourse hedges → TV toward zero), which the study reports rather than assumes.

**No new math, no `formulation.md` section.** TV is an evaluation metric computed from schedules the R2.3 program already returns; it adds no constraint, variable, or objective term. It reuses the R2.5 evaluation semantics (formulation §R2.5) and the ADR-0021 out-of-sample protocol unchanged. ## Governing reference

None (an evaluation study, standard technique; the R2.5 protocol governs). The out-of-sample commitment-then-score discipline is [ADR-0021](../decisions/0021-mpc-recourse-out-of-sample-vss.md); the honesty rule (report the distribution, do not assert the sign) is R2.5's, applied unchanged.

## Design sketch

**TV per window** = the realized-path value of the two-stage commitment fit on the **tail-augmented** scenario set, minus the realized-path value of the commitment fit on the **plain** set, both built from the same per-window forecast and residual history under the R2.1/R1.4 walk-forward.

The scoring **pattern** is R2.5's `forecast_value_from_sets`: fit each set's risk-neutral two-stage commitment, score it fixed on the realized path, return the difference; antisymmetric, exactly null when the two sets are identical (the tail-off identity). TV needs one change from that core, and it is the crux the open question below settles: `forecast_value_from_sets` settles each commitment's day-ahead leg at *its own mean*, which for the tail (whose extra trades then settle at a forecast price) structurally biases TV negative. `tail_value_from_sets` instead settles both at a shared `basis`, and the study passes `basis = realized`, so TV is the honest realized-euro difference. R2.5b's new work is that scoring variant plus the per-window harness that builds the plain and tail sets from one point forecast (and the residual-load covariate for R2.2c) and loops it, mirroring `fv_across_windows`.

## Golden oracles

Value studies get real oracles by tying to designed instances (as R2.5 does).

| # | inputs | expected | why |
|---|--------|----------|-----|
| 1 | tail set identical to plain (e.g. `γ=0` / threshold above every residual) | `TV = 0` exactly | the tail-off identity: no tail ⇒ no value difference (the R2.5-null analogue) |
| 2 | a designed window whose realized path has a spike, settled at the realized price | `TV > 0` (measured +56 at ρ=0.5 on the designed instance) | pins that the study *can* detect a real tail benefit, so a null on real data is a finding, not a blind spot |
| 3 | any window | `TV = profit_tail − profit_plain` exactly; antisymmetric under swapping the two sets | pins the bookkeeping |

## Property tests

- **Tail-off identity:** identical plain/tail sets ⇒ `TV = 0` (within solver tolerance).
- **Antisymmetry:** swapping the two sets negates `TV`.
- **Determinism:** fixed seed + inputs ⇒ bit-stable `TV`.
- **Well-formed distribution:** `tail_value_across_windows` returns one `WindowTV` per scoreable window, each with `tv = profit_tail − profit_plain`.

## Acceptance gate

*Blocks:* nothing hard; this is a measurement. It also **informed the R2.2d go/no-go.** Every box passes.

- [x] **TV distribution (headline, honest, sign not gated):** over real ENTSO-E NL (Mar–Jun 2024, 64 windows, conditional tail, verified live 2026-07-24), TV is a **null**: median **+0.00 EUR/window at every ρ** (0.1 / 0.25 / 0.5 / 1.0), means tiny (+0.03 to +0.53), 8–25% of windows positive. So representing spikes in the day-ahead scenarios does **not** change realized dispatch value on this asset, because the intraday recourse already captures a realized spike after the fact (the FV-null mechanism, confirmed for tails). Reported, not sign-asserted (R2.5 rule); the study's own designed oracle (`TV > 0`) proves the null is a real finding, not an inability to detect value. *(Token-gated integration.)*
- [x] **Golden oracles 1–3 pass exactly** (tail-off `TV=0`, designed spike-capture `TV>0` under realized settlement, bookkeeping); **property invariants pass.**
- [x] Lint/format/mypy/layers/docs-lint clean; R2.2/R2.2b/R2.2c/R2.3/R2.5 gates unchanged.

**Verdict for R2.2d: do not build.** The unconditional and conditional tails add ~0 realized dispatch value here, so a fourth tail refinement (the frequency channel `λ(x)`) would polish a representation that does not convert to euros. The honest next step is to stop refining the scenario generator.

## Out of scope

- **R2.2d (frequency-channel tail).** This study gates it; it does not build it.
- **Any optimizer / R2.3 change**, including conditioning the CVaR loss tail on the price tail (a price spike is upside for a battery, not a CVaR loss; a different object).
- **A new committed headline number.** TV is reported as a distribution with provenance, like FV; whether it earns a README claim depends on the result.

## Open questions

Posed with a proposed answer, resolved in place at review (2026-07-24). Resolved lines kept as the decision trail.

- **Day-ahead settlement basis: common forecast basis, each set's own mean, or the realized price?** **Resolved (2026-07-24, corrected during implementation): the realized day-ahead price.** The draft proposed a common *forecast* basis to control a mean-shift confound, but a designed-instance measurement showed that settling the day-ahead leg at a forecast **structurally penalizes the tail** (its extra day-ahead trades settle at the flat forecast price, worthless, while still costing degradation), so TV came out ≤ 0 almost by construction (measured −2.2 to 0 across ρ). The confound was illusory: once **both** commitments settle at the **same realized prices**, there is nothing to isolate, and TV is simply the honest "what did each commitment earn in reality." Under realized-price settlement the same designed instance gives TV = +113 at ρ=0 decaying to 0 at ρ=1, the expected shape. `basis` is still passed explicitly (not derived from a set), so the metric stays **antisymmetric** and **null on identical sets** whatever basis is used; the study passes `basis = realized`. Phase-local to the TV metric; recorded here, not an ADR.
- **Which tail is the primary comparison?** **Resolved (2026-07-24): plain (R2.2) vs conditional (R2.2c)** as the headline; the harness also accepts the unconditional (R2.2b) tail so the R2.2b-vs-plain decomposition is available.
- **Report TV across a `ρ` grid?** **Resolved (2026-07-24): yes.** `tail_value_from_sets` takes `rho`; the live study sweeps a small grid so the ρ-dependence (the FV lesson: recourse hedges, so TV likely rises as ρ tightens) is visible rather than hidden in one number.
- **Phase id.** **Resolved (2026-07-24): R2.5b** (a value-evaluation study in the R2.5 family), reusing the R2.5 scoring pattern.
